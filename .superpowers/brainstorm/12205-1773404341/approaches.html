Three Approaches

How should the event landing pages be built and where should they live?

A

Within this app — new public route /l/[slug]

Add a public-facing /l/[slug] route to BaronsHub. Point l.baronspubs.com as an additional custom domain on the same Vercel deployment. Shares the database and all existing infrastructure directly.

Pros

  • Single codebase & deployment
  • Zero data-sync complexity
  • Reuses existing event data, slug logic, venue details
  • Custom domain is a one-click Vercel config

Cons

  • Public routes live alongside admin routes (mitigated by middleware)

⭐ Recommended

B

Extend the existing short-link handler /[code]

The project already has a short_links table and /[code] route. Extend it to render landing pages instead of just redirecting — detect when a code belongs to an event and show the booking page inline.

Pros

  • Builds on existing infrastructure
  • One URL pattern for all short links

Cons

  • Overloads the redirect handler with full page rendering
  • Muddies the short-link abstraction
  • Harder to reason about as it grows
C

Separate Next.js app

A brand new, standalone Next.js project just for public event landing pages. Connects to the same Supabase database. Deployed independently to Vercel.

Pros

  • Clean separation — public site never touches admin code
  • Can be styled/branded completely independently

Cons

  • Second deployment to maintain
  • Duplicates shared types and utilities
  • More work to set up, no immediate benefit at this scale
