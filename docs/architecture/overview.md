## Auth Model

Three-layer authentication and authorisation:

### Layer 1: Middleware (`middleware.ts`)
- Supabase JWT session validation via `@supabase/ssr`
- Custom app-level session validation via `validateSession()`
- Public path bypass: `/login`, `/forgot-password`, `/reset-password`, `/auth/confirm`, `/unauthorized`, `/deactivated`, `/l` (landing pages)
- Domain rewrite: `l.baronspubs.com` routes rewritten to `/l/[slug]`
- Security headers (CSP with nonce) applied to all responses

### Layer 2: Role-Based Access Control (`src/lib/roles.ts`)
Three domain-specific roles (see [[server-actions]] for per-action checks):

| Role | Tier | Capabilities |
|------|------|-------------|
| `administrator` | Admin | Full platform access, user management, all CRUD |
| `office_worker` | Editor | Venue-scoped write (with venue_id) or global read + propose (without) |
| `executive` | Viewer | Read-only access to events, planning, reporting |

Capability functions: `canEditEvent()`, `canProposeEvents()`, `canViewArtists()`, `canCreatePlanningItems()`, `canManageVenues()`, `canManageUsers()`, `canManageSettings()`, etc.

### Layer 3: Row Level Security (Supabase RLS)
- All tables have RLS enabled
- Anon-key client used for user-scoped operations
- Service-role client used for system/cron operations only

### API Auth
- **Public API** (`/api/v1/*`): `requireWebsiteApiKey()` + `checkApiRateLimit()` via Upstash
- **Cron routes** (`/api/cron/*`): `CRON_SECRET` Bearer token
- **Session check** (`/api/auth/session-check`): Supabase `getUser()` + `validateSession()`
- **Webhooks** (`/api/webhooks/twilio-inbound`): Twilio request signature validation
