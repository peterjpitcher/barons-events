## Auth Model

Three domain-specific roles (deviation from workspace standard):

| Role | Capabilities |
|------|-------------|
| `administrator` | Full platform access, user management, all CRUD |
| `office_worker` | Venue-scoped write (if `venue_id` set) or global read-only |
| `executive` | Read-only observer across all data |

Auth flow: Middleware (Supabase JWT + custom app-session validation) -> Layout (`getCurrentUser`) -> Page -> [[server-actions]] (re-verifies auth + role checks).

API routes use separate auth: `requireWebsiteApiKey` + `checkApiRateLimit` for [[routes#Public API (v1)|public API]], `verifyCronSecret` for [[routes#Cron Routes|cron jobs]].

### Middleware Details

- **File**: `middleware.ts` (project root)
- **Matcher**: All routes except `/api/*`, `_next/static`, `_next/image`, `favicon.ico`
- **Public paths**: `/login`, `/forgot-password`, `/reset-password`, `/auth/confirm`, `/unauthorized`, `/deactivated`, `/l/*`
- **Dual-layer auth**: Supabase JWT (`getUser()`) + custom `app-session-id` cookie validated against [[data-model#app_sessions|app_sessions]] table
- **Session fixation prevention**: App session must belong to same user as Supabase JWT
- **Deactivation check**: Blocked users redirected to `/deactivated`
- **Security headers**: CSP with per-request nonce, HSTS with preload, X-Frame-Options DENY, CSRF tokens
- **Domain rewrite**: `l.baronspubs.com` requests rewritten to `/l/[slug]` for event landing pages

## Domain Model

Core entities: Events, Artists, Venues, Bookings, Customers, Debriefs, Planning Items, Planning Series, Planning Tasks, Users, Short Links, SOP Templates, Audit Log.

Event lifecycle: `draft` -> `submitted_for_review` -> `approved`/`changes_requested` -> `published` -> `completed`.

Planning system supports recurring series with task templates, SOP checklists, and AI-generated inspiration items.
