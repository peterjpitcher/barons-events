
## 2026-06-11 — "Fixed" means deployed and verified, not merged locally
Client retested the QR bug on production while the fix sat in an unmerged PR; the user had to come back with "it's not fixed yet".
**Rule:** when the user reports a production bug, the job is done only when the fix is verified live: merge → watch the production deployment succeed → smoke-test the affected path. State explicitly in the summary which environment the fix has reached. BaronsHub deploys to production from `main` via Vercel; a pushed branch deploys previews only.

## 2026-07-20 — Specs must be verified against the latest migrations and real project helpers
The venue calendar notes spec claimed the project's RLS pattern was a broad authenticated policy (true in the 2026-02 planning migration, replaced by scoped current_user_role()/current_user_venue_id() policies in 20260605143000) and named an audit helper (logAuditEvent) that exists in workspace docs but not in this project (the real helper is recordAuditLogEntry with an entity allow-list enforced in both TypeScript and a DB check constraint). It also missed multi-venue events via event_venues.
**Rule:** before writing a spec, verify every named function, table, policy and pattern against the newest migration touching that table and the actual source file, not older migrations or workspace-level conventions. Grep for join tables (event_venues, planning_item_venues) before assuming a single FK models a relationship.

## 2026-07-21 — Verify the actual user action, not just the symptom you fixed
Reported bug: deleting a calendar note closed the panel instead of deleting. I found a real z-index/stacking bug (ConfirmDialog inline at z-50 behind Sheet's body portal at z-50), fixed it, added tests, and reported it fixed. It was only half the story: the click then reached the server and the delete failed anyway with `42501`, a second, unrelated bug. The user had to report the same feature broken twice.
**Rule:** when fixing a bug in a user flow, verify the END-TO-END outcome the user described ("the note is deleted"), not the intermediate mechanism you repaired ("the click now lands"). If the flow cannot be exercised (needs auth), say so explicitly rather than claiming the bug is fixed. Where a server action is involved, probe the real database with the exact query before declaring success.

## 2026-07-21 — Postgres applies RLS read policies to the NEW row of an UPDATE
A SELECT policy of `deleted_at is null` made soft delete impossible: setting `deleted_at` makes the new row fail its own read policy, so the statement is refused with `42501 new row violates row-level security policy`, with or without RETURNING. Only deletion breaks (edits keep working), which points suspicion in the wrong direction. My first diagnosis (blaming the RETURNING clause) was wrong and I shipped it before testing.
**Rule:** never put a mutable "visibility" predicate (`deleted_at is null`, `is_active`, `status <> 'archived'`) in a SELECT policy on a table whose rows transition across that predicate. Gate writes in the UPDATE policy's USING clause and filter visibility in the data layer. And confirm an RLS diagnosis by probing the live database as the real role inside a rolled-back transaction before writing the fix.
