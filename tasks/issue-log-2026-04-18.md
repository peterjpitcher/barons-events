Calendar view file: [src/components/planning/planning-calendar-view.tsx](src/components/planning/planning-calendar-view.tsx) ŌĆö the UI walks by month via `Prev`/`Next`, but is fed by the bounded query above, so months outside the window render empty.

## Issue 07 ŌĆö No duplicate-customer prevention beyond mobile match; no "you already have a booking for this event" prompt

**Status:** Open
**Priority:** Suggested HIGH ŌĆö causes duplicate customer rows on the same person with different identifiers, and duplicate bookings on the same event.
**Area:** Public booking flow ŌåÆ customer creation.
**Route:** Public booking entry point (exact path not yet confirmed ŌĆö see Q1 below).
**Screenshot:** none (conceptual).

### Observed
[src/lib/customers.ts:161-227](src/lib/customers.ts#L161) ŌĆö `upsertCustomerForBooking` dedupes **only on mobile** (`onConflict: "mobile"`). Consequences:

1. **Same person, two mobiles** ŌåÆ two customer rows. Unavoidable without email-based merge.
2. **Same person, typo in mobile (e.g. leading zero vs `+44`)** ŌåÆ handled, because we normalise to E.164 in [src/actions/bookings.ts:69](src/actions/bookings.ts#L69) via `parsePhoneNumber(..., "GB").format("E.164")`. Good.
3. **Same person, shared number changes** ŌåÆ new mobile creates a fresh customer; email is not used to reconcile.
4. **Existing booking for the same event** ŌåÆ `createBookingAction` never checks `event_bookings` for prior bookings from the same customer + event. There is no lookup and no "you're already booked ŌĆö update?" prompt. A user re-booking silently creates a second booking row.

Searches confirm: no code path for "existing booking found for this customer and event" anywhere in `src/actions/bookings.ts`, `src/lib/bookings.ts`, or `src/lib/customers.ts`.

### Expected (Peter)
> "Are we protecting from customers having multiple profiles? If someone tries to book for an event, we should be prompting them to update their existing booking rather than make another."

1. The booking flow should recognise an existing customer by at least mobile (already done) and ideally email as a secondary key.
2. When the matched customer already has a `confirmed` booking on the same event, the UI should show a prompt: "You already have a booking for this event ŌĆö update it or cancel" ŌĆö not silently create another row.

### Suggested fix direction (not prescriptive)

**Duplicate-booking detection (the main gap):**
- In `createBookingAction`, after mobile normalisation and *before* inserting the new booking, look up:
  ```sql
  SELECT id, status, ticket_count FROM event_bookings
  JOIN customers ON customers.id = event_bookings.customer_id
  WHERE customers.mobile = :mobile
    AND event_bookings.event_id = :event_id
    AND event_bookings.status = 'confirmed';
  ```

## Issue 08 ŌĆö `/debriefs/[eventId]` is only a submission form; there is no read-only financial review and labour hours are not integrated into any post-submission summary

**Status:** Open (bug + recommendation)
**Priority:** Suggested MEDIUM ŌĆö labour hours are captured correctly; the gap is visibility after submission.
**Area:** Debriefs detail page.
**Route:** `/debriefs/[eventId]`.
**Screenshot:** none.

### What's working

- Labour hours **are** captured by the form ŌĆö [src/components/events/debrief-form.tsx:201-216](src/components/events/debrief-form.tsx#L201). The form shows a live-computed labour cost beneath the field.
- Labour hours **are** persisted ŌĆö [src/actions/debriefs.ts:115-122](src/actions/debriefs.ts#L115) reads the current `business_settings.labour_rate_gbp` and stores both `labour_hours` and `labour_rate_gbp_at_submit` on the debrief row (so the historic cost is never recomputed at a drifted rate).
- The SLT debrief email **does** include labour ŌĆö [src/lib/notifications.ts:817-819](src/lib/notifications.ts#L817) renders `"Labour: {hours}h at ┬Ż{rate_at_submit}/hr ŌĆö ┬Ż{cost}"`.

### What's broken / missing

**1. `/debriefs/[eventId]` is a submission form, not a review page.**
[src/app/debriefs/[eventId]/page.tsx](src/app/debriefs/[eventId]/page.tsx) always renders `<DebriefForm>`. There is no read-only "financial summary" card. If a debrief has already been submitted, the page simply pre-populates the form with existing values ŌĆö there is no post-submission read view that shows the financial outcome cleanly.

**2. The form's live labour cost uses the *current* rate, not the stored historic rate.**
[src/components/events/debrief-form.tsx:56-60](src/components/events/debrief-form.tsx#L56) computes `labourCost = hours * labourRateGbp` where `labourRateGbp` is the rate loaded from `business_settings` **now**. For an already-submitted debrief, the display diverges from `labour_rate_gbp_at_submit`. This is misleading ŌĆö the actual stored cost (which is what the SLT email reports) and the UI cost disagree if the rate has changed since submission.

**3. No financial summary on the event detail page either.**
A grep for `labour_hours`, `wet_takings`, `food_takings` under `src/app/events/` returns no hits. `/events/[eventId]` does not surface a post-event financial summary at all ŌĆö even though debriefs are stored against events and the data is available. This is arguably the bigger gap: the event detail page is the obvious place to show "how did this event perform".

### Expected

After a debrief is submitted, both `/debriefs/[eventId]` and `/events/[eventId]` should show a **read-only financial summary card** with:

- Attendance (+ baseline delta if `baseline_attendance` is present)
- Wet takings (+ baseline delta)
- Food takings (+ baseline delta)
- **Labour hours ├Ś stored rate (`labour_rate_gbp_at_submit`) = labour cost** ŌĆö always use the stored rate, never recompute from the current `business_settings.labour_rate_gbp`
- Total labour cost displayed as a distinct line
- A contribution line if both takings and labour are present: `(wet + food) ŌłÆ labour`
- A small "submitted by / at" footer

If the user is allowed to edit, an "Edit debrief" button takes them to the form. The form itself stays ŌĆö it's the review that's missing.

### Recommendations (for fix agent)

1. **Split `/debriefs/[eventId]` into two states:**
   - If no debrief exists ŌåÆ render the form (current behaviour).
   - If a debrief exists ŌåÆ render a `<DebriefSummaryCard>` above the form. Keep the form as a collapsible "Edit this debrief" section beneath, pre-populated as today. This lets the same page be both review and edit without a new route.

2. **Build `<DebriefSummaryCard>` as a shared component** ŌĆö reusable on both `/debriefs/[eventId]` and the financial section of `/events/[eventId]`. Component shape:
   ```ts
   type DebriefSummaryCardProps = { debrief: DebriefRow; event: EventRow; };
   ```
   It derives labour cost from `debrief.labour_rate_gbp_at_submit`, **never** from live business settings.

3. **Fix the form's live-cost display for submitted debriefs.** When `defaults?.labour_rate_gbp_at_submit` is set, the form should display that rate (and a small note "rate at submission") rather than the current rate. Only use the current rate for debriefs being drafted fresh.

4. **Add the summary card to `/events/[eventId]`** as part of a "Financials" block below the event header (exact placement matches the right-hand-rail pattern already chosen for attachments in Issue 01 ŌĆö this block can sit below the attachments rail or share the rail, design call).
