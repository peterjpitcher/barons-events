- [src/lib/planning/index.ts:578-579](src/lib/planning/index.ts#L578) — `events` `.gte("start_at", startLowerIso).lte("start_at", startUpperIso)`

Additional status filters that also violate the "without exception" requirement:

- [src/lib/planning/index.ts:556](src/lib/planning/index.ts#L556) — `planning_items` default hides `done` and `cancelled`.
- [src/lib/planning/index.ts:580](src/lib/planning/index.ts#L580) — `events` hides `completed` and `rejected`.

Calendar view file: [src/components/planning/planning-calendar-view.tsx](src/components/planning/planning-calendar-view.tsx) — the UI walks by month via `Prev`/`Next`, but is fed by the bounded query above, so months outside the window render empty.

## Issue 08 — `/debriefs/[eventId]` is only a submission form; there is no read-only financial review and labour hours are not integrated into any post-submission summary

**Status:** Open (bug + recommendation)
**Priority:** Suggested MEDIUM — labour hours are captured correctly; the gap is visibility after submission.
**Area:** Debriefs detail page.
**Route:** `/debriefs/[eventId]`.
**Screenshot:** none.

### What's working

- Labour hours **are** captured by the form — [src/components/events/debrief-form.tsx:201-216](src/components/events/debrief-form.tsx#L201). The form shows a live-computed labour cost beneath the field.
- Labour hours **are** persisted — [src/actions/debriefs.ts:115-122](src/actions/debriefs.ts#L115) reads the current `business_settings.labour_rate_gbp` and stores both `labour_hours` and `labour_rate_gbp_at_submit` on the debrief row (so the historic cost is never recomputed at a drifted rate).
- The SLT debrief email **does** include labour — [src/lib/notifications.ts:817-819](src/lib/notifications.ts#L817) renders `"Labour: {hours}h at £{rate_at_submit}/hr — £{cost}"`.

### What's broken / missing

**1. `/debriefs/[eventId]` is a submission form, not a review page.**
[src/app/debriefs/[eventId]/page.tsx](src/app/debriefs/[eventId]/page.tsx) always renders `<DebriefForm>`. There is no read-only "financial summary" card. If a debrief has already been submitted, the page simply pre-populates the form with existing values — there is no post-submission read view that shows the financial outcome cleanly.

**2. The form's live labour cost uses the *current* rate, not the stored historic rate.**
[src/components/events/debrief-form.tsx:56-60](src/components/events/debrief-form.tsx#L56) computes `labourCost = hours * labourRateGbp` where `labourRateGbp` is the rate loaded from `business_settings` **now**. For an already-submitted debrief, the display diverges from `labour_rate_gbp_at_submit`. This is misleading — the actual stored cost (which is what the SLT email reports) and the UI cost disagree if the rate has changed since submission.

**3. No financial summary on the event detail page either.**
A grep for `labour_hours`, `wet_takings`, `food_takings` under `src/app/events/` returns no hits. `/events/[eventId]` does not surface a post-event financial summary at all — even though debriefs are stored against events and the data is available. This is arguably the bigger gap: the event detail page is the obvious place to show "how did this event perform".

### Expected

After a debrief is submitted, both `/debriefs/[eventId]` and `/events/[eventId]` should show a **read-only financial summary card** with:

- Attendance (+ baseline delta if `baseline_attendance` is present)
- Wet takings (+ baseline delta)
- Food takings (+ baseline delta)
- **Labour hours × stored rate (`labour_rate_gbp_at_submit`) = labour cost** — always use the stored rate, never recompute from the current `business_settings.labour_rate_gbp`
- Total labour cost displayed as a distinct line
- A contribution line if both takings and labour are present: `(wet + food) − labour`
- A small "submitted by / at" footer

If the user is allowed to edit, an "Edit debrief" button takes them to the form. The form itself stays — it's the review that's missing.

1. **Split `/debriefs/[eventId]` into two states:**
   - If no debrief exists → render the form (current behaviour).
   - If a debrief exists → render a `<DebriefSummaryCard>` above the form. Keep the form as a collapsible "Edit this debrief" section beneath, pre-populated as today. This lets the same page be both review and edit without a new route.

   It derives labour cost from `debrief.labour_rate_gbp_at_submit`, **never** from live business settings.

3. **Fix the form's live-cost display for submitted debriefs.** When `defaults?.labour_rate_gbp_at_submit` is set, the form should display that rate (and a small note "rate at submission") rather than the current rate. Only use the current rate for debriefs being drafted fresh.
