## Issue 07 — No duplicate-customer prevention beyond mobile match; no "you already have a booking for this event" prompt

**Status:** Open
**Priority:** Suggested HIGH — causes duplicate customer rows on the same person with different identifiers, and duplicate bookings on the same event.
**Area:** Public booking flow → customer creation.
**Route:** Public booking entry point (exact path not yet confirmed — see Q1 below).
**Screenshot:** none (conceptual).

### Observed
[src/lib/customers.ts:161-227](src/lib/customers.ts#L161) — `upsertCustomerForBooking` dedupes **only on mobile** (`onConflict: "mobile"`). Consequences:

1. **Same person, two mobiles** → two customer rows. Unavoidable without email-based merge.
2. **Same person, typo in mobile (e.g. leading zero vs `+44`)** → handled, because we normalise to E.164 in [src/actions/bookings.ts:69](src/actions/bookings.ts#L69) via `parsePhoneNumber(..., "GB").format("E.164")`. Good.
3. **Same person, shared number changes** → new mobile creates a fresh customer; email is not used to reconcile.
4. **Existing booking for the same event** → `createBookingAction` never checks `event_bookings` for prior bookings from the same customer + event. There is no lookup and no "you're already booked — update?" prompt. A user re-booking silently creates a second booking row.

Searches confirm: no code path for "existing booking found for this customer and event" anywhere in `src/actions/bookings.ts`, `src/lib/bookings.ts`, or `src/lib/customers.ts`.

### Expected (Peter)
> "Are we protecting from customers having multiple profiles? If someone tries to book for an event, we should be prompting them to update their existing booking rather than make another."

1. The booking flow should recognise an existing customer by at least mobile (already done) and ideally email as a secondary key.
2. When the matched customer already has a `confirmed` booking on the same event, the UI should show a prompt: "You already have a booking for this event — update it or cancel" — not silently create another row.

### Suggested fix direction (not prescriptive)

**Duplicate-booking detection (the main gap):**
- In `createBookingAction`, after mobile normalisation and *before* inserting the new booking, look up:
  ```sql
  SELECT id, status, ticket_count FROM event_bookings
  JOIN customers ON customers.id = event_bookings.customer_id
  WHERE customers.mobile = :mobile
    AND event_bookings.event_id = :event_id
    AND event_bookings.status = 'confirmed';
  ```
- If found, return `{ success: false, error: "EXISTING_BOOKING", existingBookingId, existingTicketCount }` and let the UI surface the "update / cancel / new tickets" choice.
- If not found, proceed to insert.

**Email-based dedup (secondary):**
- When `email` is provided, run a secondary lookup: `SELECT id, mobile FROM customers WHERE email = :email`.
- If found but `mobile ≠ stored mobile`, surface an admin-only merge candidate (new table or flag) — don't auto-merge silently because that's data-loss risk.

**UI surfaces to add:**
- Booking form (public-facing): handle the `EXISTING_BOOKING` error variant with an inline prompt + action buttons.
- Admin customers page: surface a "potential duplicates" lane driven by the email-based mismatch check.

### Acceptance criteria (for fix agent)
- [ ] Booking the same event twice with the same mobile cleanly surfaces "existing booking — update or cancel" rather than creating a duplicate row.
- [ ] Updating via the prompt modifies the existing booking's ticket count (or marketing opt-in), not adds a new row.
- [ ] Email-only collisions produce a "potential duplicate" flag for admin review — no silent merge.
- [ ] Audit rows cover: existing booking found, update applied, potential duplicate flagged.
- [ ] RLS unaffected — the dedup lookup runs service-role inside `createBookingAction`.
- [ ] Test: attempt same-mobile same-event booking twice → second attempt returns the `EXISTING_BOOKING` response shape.

### Design decisions (answered by Peter 2026-04-18)
- **UX on existing-booking hit:** the form shows "We already have a booking for you for X people — would you like to change it to Y people?" — i.e. auto-update with confirmation, pre-filled with the **existing** ticket count (X) and the **requested** ticket count (Y) side by side.
