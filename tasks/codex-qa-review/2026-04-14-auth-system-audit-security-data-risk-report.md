**Findings**
1. `High` Reviewer access is broader than the spec claims. The live RLS policy grants all reviewers global event `SELECT`: `public.current_user_role() in ('central_planner', 'reviewer', 'executive')` in [20260410120003_venue_manager_event_visibility.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260410120003_venue_manager_event_visibility.sql:9). The detail page then loads the event with no assignment gate [src/app/events/[eventId]/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/[eventId]/page.tsx:75) and renders sensitive fields like `event.notes`, `event.wet_promo`, `event.food_promo`, and `event.cost_total` [src/components/events/event-detail-summary.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/events/event-detail-summary.tsx:99). This contradicts the spec’s “reviewers see assigned events once submitted” claim and exposes draft/unassigned event data to reviewers.

2. `Medium` The public landing page performs an unauthenticated service-role read of non-public events before checking public status. `getEventBySlug()` uses `createSupabaseAdminClient()` and filters only by `seo_slug` and `deleted_at` [src/app/l/[slug]/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/page.tsx:55), while the public-state check happens later in the page body [src/app/l/[slug]/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/page.tsx:121). `generateMetadata()` also builds title/description/image from that fetched row before any `approved/completed` or `booking_enabled` check [src/app/l/[slug]/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/page.tsx:100). At minimum this is a public service-role read path the spec missed; depending on Next.js render behavior, it can leak metadata for non-public events.

3. `Medium` `x-user-id` is client-spoofable. `getCurrentUser()` trusts `headers().get("x-user-id")` before calling `supabase.auth.getUser()` [src/lib/auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:48), but middleware forwards the incoming headers into `NextResponse.next()` before it later sets the “verified” user id [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:148), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:285). I verified this behavior directly: mutating the `Headers` object after `NextResponse.next()` still preserved the original client value (`attacker`). Non-planners mostly fail closed because `users` RLS only allows self-select [20250218000000_initial_mvp.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250218000000_initial_mvp.sql:162), but a `central_planner` can impersonate another app user in app-layer checks and audit attribution.

4. `Medium` The spec understates the service-role/RLS-bypass surface in the website API. It is not just `/api/v1/events`; these handlers also instantiate `createSupabaseAdminClient()`: [src/app/api/v1/events/[eventId]/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/[eventId]/route.ts:29), [src/app/api/v1/events/by-slug/[slug]/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/by-slug/[slug]/route.ts:38), [src/app/api/v1/event-types/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/event-types/route.ts:16), [src/app/api/v1/venues/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/venues/route.ts:16), and [src/app/api/v1/opening-times/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/opening-times/route.ts:62). Current filters prevent a direct draft leak in `/api/v1/events*`, but the spec’s current description materially understates how much data relies on app-layer filtering instead of RLS.

5. `Medium` Several server actions reflect backend/internal error text to the user. Examples: `return { success: false, message: \`Could not save the draft: ${detail.slice(0, 120)}\` }` in [src/actions/events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:923) and `message: error instanceof Error ? error.message : "Could not create section."` in [src/actions/sop.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/sop.ts:126). Lower-level helpers also throw raw Supabase/PostgREST messages such as `throw new Error(\`Could not update user: ${error.message}\`)` in [src/lib/users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/users.ts:82). That can expose policy names, column names, or backend internals; the spec’s blanket “generic errors across all auth paths” is too broad.

6. `Medium` The audit log contains more PII than “hashed emails only.” `logAuthEvent()` stores `ip_address` and `user_agent` alongside `email_hash` [src/lib/audit-log.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/audit-log.ts:93). The spec’s current “no plaintext PII in audit records” statement is inaccurate.

7. `Low` `CRON_SECRET` is not checked with a constant-time comparison. All cron routes use plain string equality, e.g. `if (!cronSecret || authHeader !== \`Bearer ${cronSecret}\`)` in [src/app/api/cron/cleanup-auth/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/cleanup-auth/route.ts:13), [refresh-inspiration](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/refresh-inspiration/route.ts:6), [sms-reminders](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/sms-reminders/route.ts:14), and [sms-post-event](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/sms-post-event/route.ts:14). The spec’s constant-time claim is wrong.

8. `Low` The spec overstates cookie hardening. The custom `app-session-id` cookie is `httpOnly: true` and `sameSite: "strict"` [src/lib/auth/session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:22), but Supabase SSR defaults are `sameSite: "lax"` and `httpOnly: false` [node_modules/@supabase/ssr/src/utils/constants.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/node_modules/@supabase/ssr/src/utils/constants.ts:3), and this app does not override them in [src/lib/supabase/server.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/server.ts:10). So the blanket “session cookies are httpOnly/strict” statement is only true for the custom app cookie, not the Supabase auth cookies.

9. `Low` Not all mutation inputs are runtime-validated with Zod before DB writes. `upsertVenueOpeningHoursAction()` parses `rows` from JSON and passes them straight through [src/actions/opening-hours.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/opening-hours.ts:126), and `upsertMultiVenueOpeningHoursAction()` accepts `venueIds` and `rows` with no runtime schema at all [src/actions/opening-hours.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/opening-hours.ts:149). I did not find SQL injection from this because DB access is still via Supabase query builder/RPCs, but the spec’s “all inputs Zod-validated” claim is inaccurate.

**Confirmed Checks**
- I did not find a current `/api` route with no auth at all. `/api/auth/session-check` validates both Supabase `getUser()` and `app-session-id` [src/app/api/auth/session-check/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/auth/session-check/route.ts:26), `/api/v1/*` require the website API key [src/lib/public-api/auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/public-api/auth.ts:81), and cron routes require `CRON_SECRET`.
- I did not find a path for a `venue_manager` to change roles. `updateUserAction()` is planner-only and schema-restricted [src/actions/users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:24).
- I did not find a privileged server action missing auth/role checks; the intentionally public exceptions are the auth actions and booking creation flow.
- I did not find `SUPABASE_SERVICE_ROLE_KEY` imported into client code. The admin client is guarded by `import "server-only"` [src/lib/supabase/admin.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/admin.ts:1), and I found no client-side import of it.
- I did not find token storage in `localStorage`; localStorage usage is UI-state only, and the browser Supabase client is configured via `@supabase/ssr` [src/lib/supabase/client.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/client.ts:12).
- I did not find raw SQL string concatenation or a current raw HTML sink such as `dangerouslySetInnerHTML`. The injection risk here is validation gaps and over-broad data access, not classic SQLi/XSS primitives.
- The `/api/v1/events*` handlers themselves do filter to `approved/completed` and `deleted_at is null` [src/app/api/v1/events/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/route.ts:94), so I did not confirm a direct draft leak there. The stronger public exposure is the `/l/[slug]` path above.
