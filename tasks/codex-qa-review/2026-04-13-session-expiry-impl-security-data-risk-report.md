**Findings**

1. `Medium` Supabase sign-out cookie clears do not propagate to the returned redirect responses. In [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:182), the Supabase cookie adapter writes to `res` via `res.cookies.set(...)`, but the `session_missing`, `session_expired`, and `session_mismatch` branches call `signOut({ scope: "local" })` and then return a different `redirectRes` object from [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:220), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:236), and [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:255). In local `next@16.1.5`, `NextResponse.redirect()` creates a fresh response from its own headers and status only; it does not copy cookies from an existing response instance ([response.js](/Users/peterpitcher/Cursor/BARONS-BaronsHub/node_modules/next/dist/server/web/spec-extension/response.js:98)). So there is no automatic transfer path here. Result: Supabase auth cookies can survive the app-session failure flow, leaving auth state desynchronized and potentially reusable anywhere that checks Supabase auth without the app-session layer.

2. `Low` `/api/auth/session-check` is an unthrottled auth-check path outside middleware. `/api/*` is excluded at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:296), and this route does `supabase.auth.getUser()` plus DB-backed `validateSession()` with no rate limiting at [route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/auth/session-check/route.ts:6). Comparable API routes call `checkApiRateLimit()` first, e.g. [venues/route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/venues/route.ts:9). I do not see a strong session-enumeration bug here because every failure returns the same `401`/`{ valid: false }`, and the app-session IDs are high-entropy UUIDs. It is still an avoidable abuse surface for auth-service/DB load and a yes/no validity oracle for the requester’s own cookie jar.

3. `Low` Redirect sanitization is still brittle because it validates raw strings without canonicalizing encoded separators. The checks in [login/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx:21) and [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:67) reject literal `\`, `//`, and non-leading-slash inputs. Raw `javascript:` and `data:` URLs are therefore rejected. `%5C` arriving through `searchParams` is decoded before `sanitizeRedirect`, so that case is blocked. But a tampered `redirectTo` form field is validated before decoding, so `/%5C...` still passes the current check. Based on standard WHATWG URL parsing tested locally, `/%5Cevil.com` stayed same-origin, so I did not confirm an exploitable open redirect from this exact payload. The issue is that the validator depends on caller-specific decoding behavior and should normalize before validating.

4. `Informational` The `SessionMonitor` fail-open behavior is acceptable only if this component is treated as UX, not enforcement. In [session-monitor.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/session-monitor.tsx:24), network errors are swallowed and the user stays on the current page. That does not create new server-side access because middleware and server handlers remain authoritative. It does mean stale sensitive UI can remain visible longer, and if you wanted this monitor to be a hard local expiry control, the current posture is too permissive.

- If any excluded `/api/*` route authorizes on Supabase cookies alone, finding 1 becomes more severe.
