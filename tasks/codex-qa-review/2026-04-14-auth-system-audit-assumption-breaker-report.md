**Findings**
1. **Broken | High confidence.** Spec claim: “All 16 action files call `getCurrentUser()`.” There are 13 top-level runtime modules under `src/actions/*.ts`, not 16; you only get to 16 if you include the 3 test files under `src/actions/__tests__/`. More importantly, [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:66) exports four server actions with no `getCurrentUser()` at all, and [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:36) has a public write path `createBookingAction()` with no auth check; only `cancelBookingAction()` is auth-gated at [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:171). Impact: the accurate statement is “most non-public mutation paths are auth-gated, with intentional exceptions,” not blanket coverage.

2. **Broken | High confidence.** Spec claim: “No audit logging on data mutations.” Audit already exists in [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:67) (`auth.role.changed`), [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:213) (`booking.cancelled`), [debriefs.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:174) (`event.debrief_updated`), [customers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/customers.ts:66) (direct `audit_log` insert), [sop.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/sop.ts:116) and many later SOP mutations, plus multiple event mutations in [events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1276), [events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1427), and [events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1798). Impact: this is a partial-coverage gap, not a total absence; the unaudited modules are narrower: `artists`, `event-types`, `links`, `opening-hours`, `planning`, and `venues`.

3. **Partially broken | High confidence.** Spec claim: public API RLS bypass is “High” severity. The implementation does use service-role clients in all three endpoints: [list](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/route.ts:47), [detail](</Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/[eventId]/route.ts:29>), and [by slug](</Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/by-slug/[slug]/route.ts:38>). But each endpoint hard-filters to `approved|completed` and `deleted_at is null` at [route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/route.ts:94), [detail](</Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/[eventId]/route.ts:75>), and [by slug](</Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/by-slug/[slug]/route.ts:85>), and the serializer only returns public-facing fields in [public-api/events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/public-api/events.ts:175). Impact: the defense-in-depth concern is real, but the current leak surface is public event content, not arbitrary rows, drafts, bookings, or users; “High” is unearned from the current code.

4. **Confirmed | High confidence.** Spec claim: idle timeout is not enforced in middleware. [validateSession()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:79) only checks `expires_at` at lines 92-105 and never checks `last_activity_at`; middleware just trusts that result at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:235). Impact: this gap is real.

5. **Confirmed in app code | Medium-high confidence.** Spec claim: password reset relies on Turnstile only. [requestPasswordResetAction()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:207) does Turnstile at lines 223-228, then calls `admin.generateLink()` at [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:233) with no `RateLimiter`, no `login_attempts`, and no per-email/IP throttle; those helpers are only used for login in [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:175) and [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:204). Impact: there is no first-party reset throttling in this repo.

6. **Broken / needs narrowing | High confidence.** Spec claim: venue isolation is app-code only. The events table already has venue-scoped `SELECT` RLS for `venue_manager` in [20260410120003_venue_manager_event_visibility.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260410120003_venue_manager_event_visibility.sql:9), and event bookings have venue-scoped read/update RLS in [20260410120002_tighten_event_bookings_rls.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260410120002_tighten_event_bookings_rls.sql:16) and [same](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260410120002_tighten_event_bookings_rls.sql:47). What is still missing is venue-scoped write RLS on `events`; the update policy remains creator/central-planner based in [20260321000001_fix_event_update_rls.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260321000001_fix_event_update_rls.sql:8), and event writes prefer service-role first in [events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:381). Impact: the spec overstates the gap; read isolation is already DB-enforced, write isolation is the actual hole.

7. **Confirmed | High confidence.** Spec claim: session fixation protection exists. Middleware compares the Supabase JWT user to the app-session user at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:259); on mismatch it signs out locally, clears `app-session-id`, and redirects at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:269). Impact: this claim is earned.

8. **Confirmed | High confidence.** Spec claim: HIBP breach check exists. [password-policy.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/password-policy.ts:47) calls `checkHibp()`, and [password-policy.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/password-policy.ts:66) SHA-1 hashes the password and calls `https://api.pwnedpasswords.com/range/...` with `Add-Padding`. Impact: this part of the spec is correct.

9. **Confirmed with caveat | High confidence.** Spec claim: max 5 sessions per user. The cap is encoded at [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:8), eviction happens at [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:45), and there is a direct test at [session.test.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/__tests__/session.test.ts:180). Impact: the code clearly targets a 5-session cap, but it is best-effort rather than atomic, so concurrent sign-ins could exceed 5 temporarily.

10. **Confirmed | High confidence.** Spec claim: `current_user_role()` is callable by `anon`. The grant is explicit in [20250301000000_secure_current_user_role.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250301000000_secure_current_user_role.sql:14) and [same](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250301000000_secure_current_user_role.sql:15).
