**2. Auth And Session Lifecycle**

- Login is a two-step success path: [signInAction()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:66) signs into Supabase with [createSupabaseActionClient()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/server.ts:25), then creates a custom `app_sessions` row with [createSession()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:36) and sets the `app-session-id` cookie at [src/actions/auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:145). If custom session creation fails, it explicitly signs Supabase back out at [src/actions/auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:151).
- The custom session record stores `created_at`, `last_activity_at`, `expires_at`, `user_agent`, and `ip_address`; expiry is `24h` absolute and max active sessions per user is `5` at [src/lib/auth/session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:6) and [src/lib/auth/session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:45).
- Runtime validation happens in middleware, not in the browser. After public-path bypass, middleware validates the Supabase user with `supabase.auth.getUser()` at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:196), then requires an `app-session-id` cookie at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:214), then calls [validateSession()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:79), then checks `session.userId === user.id` at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:244).
- `validateSession()` is fail-closed and only checks `expires_at`; it returns `lastActivityAt` but does not enforce or refresh it anywhere in runtime code at [src/lib/auth/session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:92). The older migration still describes idle cleanup via `last_activity_at` at [20260311100000_auth_session_tables.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260311100000_auth_session_tables.sql:38), but the live runtime cleanup helper only deletes `expires_at < now` and stale `login_attempts` at [src/lib/auth/session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:144).
- Logout destroys both layers in sequence: [signOutAction()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:177) deletes the custom session row if the cookie exists, clears the cookie, logs `auth.logout`, signs out Supabase, and redirects to `/login`.
- Session revocation is also used outside logout: [updateUserAction()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:24) destroys all sessions for the target user after a role change at [src/actions/users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:58), and [completePasswordResetAction()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:265) destroys all sessions, creates a replacement `app-session-id`, then signs Supabase out at [src/actions/auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:311).

**3. Server Auth Resolution And App Shell**

- [getCurrentUser()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:48) is the main server-side identity helper. It creates a readonly Supabase server client, tries to read `x-user-id` from request headers first, otherwise falls back to `supabase.auth.getUser()`, then loads the app profile from `users` and normalizes the role.
- [getSession()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:39) exists, but its own docstring says it can be stale/revoked and must not be used for authorization. In repo search, its only runtime mentions are its definition and the warning comment in [src/app/login/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx:34).
- Middleware’s code intends to pass a verified user id downstream by mutating `requestHeaders` with `x-user-id` at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:259); [getCurrentUser()](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:51) is written to consume that header.
- The root layout is auth-aware but not auth-enforcing. [src/app/layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx:150) awaits `getCurrentUser()` and conditionally renders `<AppShell user={user}>` only when `user` is truthy at [src/app/layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx:168); otherwise it renders bare `children`.
- [AppShell](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/app-shell.tsx:65) is a server component. It filters nav items by `user.role`, renders sidebar/header chrome, and wires both sign-out buttons to the server action [signOutAction](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/app-shell.tsx:127).
- Navigation inside the shell is client-side. [NavLink](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/nav-link.tsx:14) is a `"use client"` component using Next `<Link>` plus `usePathname()`. The mobile drawer reuses the same `NavLink` in [src/components/shell/mobile-nav.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/mobile-nav.tsx:77).

**Bottom Line**

- Protected-page auth is server-gated and dual-layer: Supabase JWT first, custom `app-session-id` second.
- The app shell is server-rendered off `getCurrentUser()`, while shell navigation is client-side `<Link>` navigation.
- In the current repo, there is no client session observer, no tab-focus/session event listener, no `Suspense` boundary around auth resolution, and no single consistent expired-session behavior across server actions.
