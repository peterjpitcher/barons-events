**Findings**
- `SEC-001 Critical` Public booking creation can bypass the app entirely. `create_booking()` is a `SECURITY DEFINER` RPC with no visible execute restriction in its defining migration, while the app’s protections live in [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts#L34) and [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/bookings.ts#L36). Concrete exploit: an internet user can call Supabase RPC directly, bypass the server-action rate limit/validation, and the RPC itself does not enforce `max_tickets_per_booking`. Refs: [20260313000000_event_bookings.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000000_event_bookings.sql#L79), [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts#L34), [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/bookings.ts#L36)

- `SEC-002 Critical` Booking reminder/post-event RPCs appear callable outside cron and expose customer PII. The cron routes treat them as privileged, but their `SECURITY DEFINER` definitions show no visible `REVOKE/GRANT` hardening. Concrete exploit: an unauthenticated attacker with the public anon key can hit the RPC endpoint and pull customer first names/mobile numbers for tomorrow’s or yesterday’s bookings. Refs: [20260313000000_event_bookings.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000000_event_bookings.sql#L131), [20260313000000_event_bookings.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000000_event_bookings.sql#L159), [sms-reminders](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/sms-reminders/route.ts#L21), [sms-post-event](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/sms-post-event/route.ts#L21)

- `SEC-003 Critical` Customer export is bypassable at the database layer. `list_customers_with_stats()` is `SECURITY DEFINER`, trusts caller-supplied `p_venue_id`, and has no visible execute restriction; the app-layer scoping in [customers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/customers.ts#L30) is therefore not a backstop. Concrete exploit: a reviewer, executive, or anonymous caller can invoke the RPC with `p_venue_id = null` and dump all customer names, mobiles, emails, consent flags, and booking stats. Refs: [20260313000001_add_customers_and_consent.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000001_add_customers_and_consent.sql#L83), [customers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/customers.ts#L30)

- `SEC-004 High` Any authenticated staff user can read and mutate bookings they do not own. RLS grants all authenticated users `SELECT` and `UPDATE` on `event_bookings`, and the admin-backed helpers in [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/bookings.ts#L67) and [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts#L159) rely only on app checks; the UI also lets executives into the bookings page by omission. Concrete exploit: a reviewer can query `event_bookings` directly with their JWT and cancel any booking, or an executive can browse to `/events/<id>/bookings` and use the built-in cancel button. Refs: [20260313000000_event_bookings.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000000_event_bookings.sql#L67), [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts#L159), [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/bookings.ts#L83), [page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/[eventId]/bookings/page.tsx#L41)

- `SEC-005 High` Planning data is exposed to roles that should not have planning access, and loading `/planning` performs unauthorized writes. The page only checks “authenticated,” while planning table read policies also allow all authenticated users; the loader then uses the admin client to read all planning users/items/tasks/events and to generate recurring occurrences/SOP tasks on page load. Concrete exploit: a venue manager or reviewer visits `/planning`, harvests all planning data plus staff emails, and triggers missing planning occurrences/tasks to be created via a GET request. Refs: [page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/planning/page.tsx#L13), [20260225000001_tighten_planning_rls.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260225000001_tighten_planning_rls.sql#L11), [index.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts#L406), [index.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts#L462)

- `SEC-006 High` Planner-only SOP mutations are reachable by non-planners via exposed `SECURITY DEFINER` RPCs. `generate_sop_checklist()` and `recalculate_sop_dates()` bypass RLS, show no visible execute restriction, and are wired from admin-backed helpers; planning item IDs are readable to any authenticated user. Concrete exploit: a reviewer/executive queries planning item IDs, then calls the RPC endpoint directly to regenerate or reschedule SOP tasks for planner-owned work. Refs: [20260408120003_add_sop_rpc_functions.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120003_add_sop_rpc_functions.sql#L26), [20260408120003_add_sop_rpc_functions.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120003_add_sop_rpc_functions.sql#L241), [20260225000001_tighten_planning_rls.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260225000001_tighten_planning_rls.sql#L35), [sop.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/sop.ts#L9)

- `SEC-007 Medium` Executives are not actually read-only in planning. The role model says executive is read-only, but `dismissInspirationItemAction()` allows `canViewPlanning`, and dismissals are organization-wide. Concrete exploit: an executive hides inspiration items for every planner by inserting global dismissal rows, even though they should only observe planning. Refs: [roles.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/roles.ts#L9), [roles.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/roles.ts#L56), [planning.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/planning.ts#L610), [20260311120000_add_planning_inspiration.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260311120000_add_planning_inspiration.sql#L64), [index.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts#L425)

- `SEC-008 Medium` The website API leaks internal event notes. The v1 event routes select `notes`, and `toPublicEvent()` falls back to `notes` whenever `public_description` is missing. Concrete exploit: any bearer-key holder requests an approved/completed event with unfinished web copy and receives staff-only internal notes in the public `description` field. Refs: [events route](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/route.ts#L55), [event by id route](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/[eventId]/route.ts#L36), [public-api events](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/public-api/events.ts#L186)

- `SEC-009 Medium` Password-reset requests clear lockout state before mailbox ownership is proven. `requestPasswordResetAction()` deletes lockout records immediately, and Turnstile is fail-open on missing/unavailable tokens. Concrete exploit: an attacker can alternate failed logins with password-reset requests for the same email and keep resetting the 5-attempt lockout, materially weakening brute-force protection. Refs: [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts#L70), [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts#L232), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L214)

- `SEC-010 Medium` `/api/auth/heartbeat` renews whatever `app-session-id` cookie is presented and is explicitly CSRF-exempt. Middleware’s session-ownership check does not run on `/api/*`, and `renewSession()` updates by `session_id` alone. Concrete exploit: if a low-privilege user ever learns another user’s `app-session-id`, they can keep that victim session alive indefinitely by POSTing heartbeat with their own valid login plus the victim cookie value. Refs: [heartbeat](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/auth/heartbeat/route.ts#L12), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L139), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L244), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L280)

The RPC findings assume normal PostgreSQL default function privileges, because this repo defines those `SECURITY DEFINER` functions but never shows matching execute revokes/grants. No confirmed SQL injection bug emerged from the reviewed auth/RBAC paths.
