**Integration & Architecture Report**

**Findings**
1. High: the proposed 4-PR phasing is neither merge-safe nor rollout-safe. Phase 1 changes `UserRole`, `auth.ts`, and the DB migration only ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:123)), but the app shell depends on `getCurrentUser()` at runtime ([layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx:164)), and `normalizeRole()` rejects any non-legacy role string ([auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:18)). If the DB migration runs before the new code, users are treated as unauthenticated. Separately, PR 1 cannot compile cleanly as scoped because old literals remain in runtime code such as [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:17) and [app-shell.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/app-shell.tsx:21), and PR 2 cannot compile cleanly as scoped because it renames/removes capability APIs that are still imported by [planning.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/planning.ts:20) and [planning/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/planning/page.tsx:6).

2. High: removing `reviewer` is not just a role rename; it breaks an existing workflow model the spec does not replace. The spec says `office_worker` replaces `reviewer` but cannot review/approve events ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:25), [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:49)), while the current system still stores `default_reviewer_id` on venues ([venues.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/venues.ts:17)), populates venue forms from `users.role = 'reviewer'` ([reviewers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/reviewers.ts:9), [venues/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/venues/page.tsx:22)), auto-assigns submitted events to a default reviewer or first reviewer ([events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1236)), and records approvals against `reviewer_id` ([events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:732), [seed.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/seed.sql:262)). That is a missing architecture decision, not cleanup.

3. High: the reversibility claim is false as written. The spec says an inverse migration can restore original role strings ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:117)), but the forward migration irreversibly merges both `venue_manager` and `reviewer` into `office_worker` ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:100)). The schema only stores `role` plus a nullable `venue_id` ([initial_mvp.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250218000000_initial_mvp.sql:33)), and user invite/update flows allow blank `venueId` for any role ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:17), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:94)). After migration, you cannot deterministically recover which `office_worker` used to be a reviewer.

4. High: the spec underestimates the authorization surface that bypasses RLS. A lot of effective RBAC lives in admin-client helpers and route loaders, not in `roles.ts`: planning board and recurrence generation use the admin client ([planning/index.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:383), [planning/index.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:478)); booking and customer helpers do the same ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/bookings.ts:63), [all-bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/all-bookings.ts:37), [customers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/customers.ts:30)); event flows rely on caller-side ownership/venue checks before service-role writes ([events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:591)). Phase 4 says “all server actions” move to capability functions ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:180)), but it does not explicitly audit `src/lib/**` helper-layer bypasses, and it does not define capabilities for bookings or customers even though the matrix changes those permissions ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:47)).

5. High: `office_worker` does not fit the current `venue_manager` architecture cleanly because the existing role mixes venue-scoped and creator-scoped behavior. Venue-wide access currently exists for bookings/customers and some event-adjacent flows via `user.venueId` ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:192), [all-bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/all-bookings.ts:56), [customers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/customers.ts:34)), but event listing, event submission, debriefs, and deletion are still creator-owned ([events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:173), [events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1150), [debriefs.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:50), [page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/debriefs/[eventId]/page.tsx:20)). The spec maps `venue_manager` to `office_worker` ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:56)) but never states whether `office_worker` retains `venue_id`, or whether current creator-owned behavior is supposed to become office-wide.

6. Medium: the consumer inventory is incomplete, so the spec overstates what typecheck/build will catch and understates out-of-scope breakage. Generated DB types still treat `users.role` as `string` ([types.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/types.ts:12)); planning people are also stringly typed ([types.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/types.ts:11)); user actions validate legacy role literals directly with Zod ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:17)); reviewer and assignable-user queries are raw string filters ([reviewers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/reviewers.ts:9), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/users.ts:139)). The spec also marks notification logic out of scope ([spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:209)), but notification recipients are hardcoded to `central_planner` today ([notifications.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:629), [notifications.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:711)).

**Checks**
- `auth.ts` is the real blast-radius hub; `roles.ts` is not. The critical path is `layout -> getCurrentUser -> normalizeRole`, while many runtime permissions still bypass `roles.ts`.
- `public.users.role` is the effective source of truth. `current_user_role()` falls back to `auth.jwt()->>'role'` ([secure_current_user_role.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250301000000_secure_current_user_role.sql:1)), but I found no production code that updates auth role metadata. The migration must update both the helper and every direct `users.role` consumer.
- DB policy styles are mixed exactly as you flagged: some policies use `current_user_role()`, others query `public.users.role` directly ([venue_opening_times.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260227000001_venue_opening_times.sql:65), [short_links.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260228000003_short_links.sql:41), [add_customers_and_consent.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000001_add_customers_and_consent.sql:43)). I found no SQL views/materialized views in `supabase/migrations`, so there is no frozen-view-column issue.
- Deployment ordering risk is real. As specified, DB-first rollout can lock out users before the app code is live.
- `venue_id` remains a structural dependency unless the design explicitly removes venue-scoped bookings/customers access. The current codebase assumes a venue-linked staff role in multiple domains.
