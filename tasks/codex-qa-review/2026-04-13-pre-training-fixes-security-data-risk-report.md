**Findings**
- Low: `cancelBookingAction` now fixes caller-supplied `eventId` spoofing, but it introduces a small booking-existence oracle because the service-role lookup runs before authorization. [src/actions/bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:180) does the admin lookup first, and only later rejects disallowed roles at [src/actions/bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:191). An authenticated `reviewer`/`executive`, or a `venue_manager` probing another venue, can distinguish `Booking not found` from `You do not have permission` / `You can only cancel...` and confirm whether a booking UUID exists. Recommended fix: reject non-`central_planner`/`venue_manager` before any admin lookup, normalize `not found` vs `out of scope` errors for scoped users, and remove the now-unused `eventId` parameter to avoid future trust regressions.

- Low: The draft-review vuln is fixed in the server action, but the UI and tests are not aligned with the new rule. [src/actions/events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1351) now correctly rejects anything outside `submitted`/`needs_revisions`, yet the board and planning overlay still render approve controls for drafts at [src/components/events/events-board.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/events/events-board.tsx:165) and [src/components/planning/planning-item-card.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-item-card.tsx:791). There is also no direct `reviewerDecisionAction` regression test. Today this fails closed, but it keeps the old exploit path visible and makes reintroduction easier. Recommended fix: update both UI predicates to match the action and add explicit tests that drafts are rejected while `submitted`/`needs_revisions` are allowed.
