# Security & Data Risk Review: RBAC Renovation Spec

**Scope:** [RBAC renovation spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:1)

## Findings

1. `SEC-001 High` The spec treats a role merge as a string rename, but the live policies encode two different security scopes. `office_worker` replaces both `venue_manager` and `reviewer`, and the spec says old role literals should just be updated in-place. In the live event RLS, `reviewer` is on the global-read branch while `venue_manager` is venue-scoped; a naive rename makes every `office_worker` match the broader branch. That means the spec removes `canReviewEvents`, but it does not remove the effective `canViewAllEvents` behavior for former venue managers. Exploit: a former `venue_manager` logs in as `office_worker` and reads other venues’ events and debriefs. Refs: [role model spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:25), [route access spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:89), [RLS update spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:108), [events RLS](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260410120003_venue_manager_event_visibility.sql:9), [current event scoping](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:164), [debrief visibility policy](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250315090000_events_assignee_refactor.sql:89).

2. `SEC-002 High` The planning-board change is a real data-exposure change because that page already bypasses RLS with the service-role client. The spec grants `office_worker` access to the planning board and says they can reach read-only event details from there. The current loader fetches all planning items, all event overlays in the window, and all users with emails through `createSupabaseAdminClient()`. The event detail page also loads linked planning tasks through the admin client whenever `canViewPlanning()` passes. Exploit: a former venue manager becomes `office_worker`, opens `/planning`, and gets org-wide event titles, statuses, venue data, assignee names, and staff emails. Refs: [planning access spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:38), [planning nav spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:73), [planning page](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/planning/page.tsx:19), [planning users loader](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:406), [planning board loader](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:478), [event detail planning fetch](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/[eventId]/page.tsx:116), [admin client warning](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/admin.ts:6).

3. `SEC-003 High` The new debrief model is not enforceable with the current schema and write path. The spec opens debrief read/create to all roles and limits edits to `administrator` plus creator, but the live implementation is a single `upsert` on `event_id` that rewrites `submitted_by` on every save. There is no immutable debrief owner column and no DB policy tied to debrief authorship. If RLS is loosened so non-admins can “create,” they also gain an overwrite path for existing rows, and the overwrite changes who the system thinks the creator is. The spec is also internally inconsistent for `executive`: “create/read = yes” but “edit own = no” cannot be implemented on an upsert-based form. Exploit: any authenticated user with debrief submit access overwrites someone else’s takings/operational notes and becomes the new recorded submitter. Refs: [debrief spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:35), [debrief route spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:91), [creator-edit spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:145), [debrief action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:99), [debrief upsert](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/debriefs.ts:27), [debrief table/policies](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250218000000_initial_mvp.sql:98), [debrief sensitive fields](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260210121000_expand_debriefs_and_event_images.sql:4).

4. `SEC-004 Medium` JWT/session migration is underspecified and can create split-brain authorization. The app mostly fail-closes because `getCurrentUser()` reads `public.users.role`, not the JWT role, but SQL still uses `current_user_role()` and that function falls back to `auth.jwt()->>'role'`. The spec changes DB role strings and `normalizeRole()` but does not require forced reauth or session revocation, even though ordinary role changes already destroy sessions. If any old-role SQL check survives, or a user hits the JWT fallback path, stale claims can keep old privileges alive in SQL while the app thinks the user has the new role. Refs: [migration spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:98), [phase 1 spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:123), [risk table spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:198), [getCurrentUser](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:61), [current_user_role](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250301000000_secure_current_user_role.sql:1), [existing session-destroy behavior](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:65).

5. `SEC-005 Medium` Seed/bootstrap sequencing is internally inconsistent and will break environment resets during the migration window. The spec changes the role constraint in phase 1, but `seed.sql` still inserts `central_planner`, `reviewer`, and `venue_manager`, and the spec separately defers seed cleanup to phase 4. `supabase db reset` applies all migrations and then runs the seed, so the seeded users will violate the new constraint immediately after the migration lands. That is a real data-integrity and verification risk for an auth migration. Refs: [seed update spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:114), [phase 4 seed step](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:186), [seed users](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/seed.sql:88), [import actor lookup](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260206120000_import_baronspubs_2026_events.sql:21), [planning seed lookup](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120007_seed_planning_calendar_2026_2027.sql:10).

## Open Questions

1. Is `office_worker` supposed to have global cross-venue read, or venue-scoped read? The current 3-role design cannot safely represent both former `reviewer` and former `venue_manager` scopes without an extra attribute or a fourth role.
2. Should `executive` really be allowed to create debriefs? If not, the capability matrix and route-access section need to be tightened before implementation.
3. The spec’s RLS verification plan relies on grepping migration history. For a live auth migration, the safer check is the active schema (`pg_policies`, functions, triggers), because merged-role policies need redesign, not literal replacement.

## Bottom Line

This spec is not safe to implement as written. The biggest problem is that it collapses two materially different trust boundaries into `office_worker` and then describes the DB work like a rename. That creates clear cross-venue read paths through both RLS and service-role loaders, and the debrief rules are currently unenforceable at the database boundary.

No code changes were made.
