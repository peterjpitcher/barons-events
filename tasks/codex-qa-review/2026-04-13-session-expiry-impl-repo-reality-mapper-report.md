The adapter writes to `res.cookies` [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:187), but the failure branches return fresh `redirectRes` objects [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:226), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:242), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:261).

**Risk Areas**
- Critical: the redirect-loop fix is probably incomplete because the Supabase cookie-clearing side effect is written to a response object that is not returned. If those cookies remain, [src/app/login/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx:41) still redirects away on any valid Supabase user, so the core RC7 loop can persist.
- High: `SessionMonitor` only preserves `window.location.pathname` [src/components/shell/session-monitor.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/session-monitor.tsx:20). Middleware preserves `pathname + search` [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:224), so query-string state is lost on the client-driven expiry path.
- High: the route handler is a probe, not a faithful mirror. Because it discards cookie writes, token refresh behavior can differ from middleware, especially around expired access tokens with still-valid refresh tokens.
- Medium: the implementation closes refocus/BFCache cases, but not the full success-criteria surface in the spec. Active-tab expiry, nav-click-before-refocus, and server-action submit behavior are still largely untouched [tasks/session-expiry-flash/spec.md](/Users/peterpitcher/Cursor/BARONS-BaronsHub/tasks/session-expiry-flash/spec.md:190).
