- RLS-only remediation is not enough for this repo. The risky paths use the admin client in [`bookings.ts`](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/bookings.ts#L67), [`planning/index.ts`](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts#L478), and [`events/[eventId]/page.tsx`](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/[eventId]/page.tsx#L116). Repo-fit remediation needs three changes together: gate [`/planning`](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/planning/page.tsx#L13), remove non-view roles from the nav, and close the event-detail planning/SOP leak. [`app-shell.tsx`](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/shell/app-shell.tsx#L33)

- The booking CAPTCHA fix should reuse the auth pattern, but `verifyTurnstile()` is private to [`auth.ts`](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts#L70). Repo-fit means extracting a shared helper or duplicating the logic intentionally, then wiring the public form to submit the token.

Bottom line: Claude got the headline bugs on booking cancellation, public booking bot protection, and `/planning` gating, but the spec is overconfident on lockout casing, underestimates how often this repo bypasses RLS with the admin client, and misses the strongest issues around silent audit-log failure, direct planning mutations by executives, password-reset lockout clearing, and unhardened `SECURITY DEFINER` RPCs.
