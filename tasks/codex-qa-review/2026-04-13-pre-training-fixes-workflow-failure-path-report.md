**Findings**
- High: Newly created opening overrides cannot be safely edited before a refresh. The server action drops the real inserted ID instead of returning it ([opening-hours.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/opening-hours.ts:209)), so the client fabricates one with `crypto.randomUUID()` for optimistic state ([overrides-calendar.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/opening-hours/overrides-calendar.tsx:458)). The next edit submits that fake ID, and `updateOpeningOverride` treats a 0-row update as success because it only checks `error`, not whether anything matched ([opening-hours.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/opening-hours.ts:250)). Demo symptom: “Override updated” toast, then the change disappears after reload.

- High: `reviewerDecisionAction` can report success on a stale tab or double-submit without updating the event row. It tries to guard reviewer updates with `assignee_id = user.id`, but the shared updater only checks `error` and never verifies affected rows ([events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:381)). If the event was already reassigned or another decision landed first, the update becomes a silent no-op; the action still records an approval, audit/version entries, and sends email ([events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1408), [events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1458)). Demo symptom: “Decision recorded” even though the event status on refresh is unchanged or inconsistent with the audit trail.

- Medium: Opening-hours edits from a filtered venue view clobber the optimistic cache for other venues. The page shell derives `filteredOverrides` from the full list ([opening-hours-page-shell.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/opening-hours/opening-hours-page-shell.tsx:46)) but passes `setLocalOverrides` straight into the calendar as if the child had the full dataset ([opening-hours-page-shell.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/opening-hours/opening-hours-page-shell.tsx:201)). The calendar writes back from its filtered `overrides` prop on delete/save ([overrides-calendar.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/opening-hours/overrides-calendar.tsx:145), [overrides-calendar.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/opening-hours/overrides-calendar.tsx:355)). Demo symptom: filter to one venue, make a change, switch back to all venues, and unrelated overrides disappear until reload.

- Medium: The `needs_revisions` fix is only partial across demo entry points. The server action now accepts `needs_revisions` for review ([events.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1351)), but quick-review surfaces still hard-code `["submitted", "draft"]` ([events-board.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/events/events-board.tsx:163), [planning-item-card.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-item-card.tsx:791)). Demo symptom: a `needs_revisions` event looks “stuck” from the calendar/planning views even though the detail page can re-review it.
