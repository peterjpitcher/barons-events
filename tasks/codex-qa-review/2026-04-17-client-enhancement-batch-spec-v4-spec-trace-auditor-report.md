## Open Questions

- Should `create_multi_venue_event_proposals` move after the Wave 3 event-status/nullability migration, or should the relaxation move earlier?
- Should `preApproveEventAction` create the planning item and SOP checklist in one transaction before transitioning to `approved_pending_details`?
- Should no-venue office workers be read-only everywhere? The spec says yes for attachments but not in multi-venue RPC/status-trigger SQL.
- Should per-venue child tasks inherit dependencies/blocked state from their master, or is the weaker child behaviour intentional?
- Should trigger functions receive `GRANT EXECUTE TO service_role`, or should the global SECURITY DEFINER convention explicitly distinguish trigger functions from direct-call RPCs?

## Recommendation

**V5 NEEDED.**

Do not implement from v4 as written. The remaining blockers are not cosmetic: proposal RPC ordering, missing approval-time planning-item creation, SQL-level authorisation gaps in multi-venue RPCs, no-venue office-worker leakage in the status trigger, and the incomplete audit enum all need one more spec pass before implementation.
