**Workflow & Failure-Path Report**

**Findings**
- High: Turnstile is still fail-open on the public booking flow. The client submits `undefined` whenever the widget has not injected `cf-turnstile-response` yet ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:55)), the booking schema makes `turnstileToken` optional ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:19)), and `verifyTurnstile()` returns `true` for missing tokens, missing secret, siteverify outages, and thrown fetches ([turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:6)). Result: scenario 2, scenario 5, and most of scenario 7 still succeed without CAPTCHA rather than degrading closed.

- High: Rapid repeated submits can create duplicate confirmed bookings. `handleSubmit()` has no synchronous re-entry guard before it calls the server action ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:50)), so double-clicks or Enter spam can race before the `loading` rerender disables the button ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:233)). Server-side protection is only an in-process per-IP limiter ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:16), [rate-limit.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/public-api/rate-limit.ts:4)) plus capacity locking in the RPC ([20260313000000_event_bookings.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000000_event_bookings.sql:95)). There is no idempotency key, no uniqueness constraint on event/mobile/email, and the RPC always inserts a new booking row ([20260313000000_event_bookings.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000000_event_bookings.sql:119)).

- Medium: Token-expiry handling has no explicit UX and can collapse into either a generic error or a silent bypass. The widget is configured with defaults only, with no `expired-callback`, `error-callback`, or readiness state in the app ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:225)). Cloudflare documents that tokens expire after 300 seconds and that `refresh-expired` defaults to `auto`, so the common case is silent refresh rather than a visible error. If an expired token still reaches `siteverify`, the user only sees `Security check failed. Please try again.` ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:53)). If the hidden input is absent during refresh, the current fail-soft path lets the booking through anyway ([turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:7)). Cloudflare refs: https://developers.cloudflare.com/turnstile/get-started/server-side-validation/ and https://developers.cloudflare.com/turnstile/get-started/client-side-rendering/widget-configurations/

- Low: `revalidate = 60` is misleading for this route as currently built. The page declares ISR ([page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/page.tsx:9)), but nonce plumbing depends on `headers()` in both the root layout ([layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx:155)) and the booking page ([page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/page.tsx:130)). In a production build, `/l/[slug]` is emitted as dynamic `ƒ`, not ISR, so stale cached nonces are not a live issue today.

**Checks**
1. User loads the page, Turnstile renders, fills the form, submits: the token read path should work in the happy case. The widget sits inside the `<form>` ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:94), [BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:226)), `handleSubmit()` looks up `[name="cf-turnstile-response"]` inside that form ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:55)), and Turnstile's default form integration uses that exact hidden field name. The token is then forwarded into `createBookingAction()` ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:59)) and validated server-side ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:51)). Caveat: this is brittle DOM coupling; if the widget config ever changes `response-field-name` or disables the response field, booking CAPTCHA silently stops being enforced.

2. User loads the page but the Turnstile CDN is down or blocked: the form still works. `afterInteractive` means the form UI can become interactive without the external script ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:243)), the DOM query returns `undefined` when the hidden field never appears ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:55)), and `verifyTurnstile()` explicitly treats missing tokens as allowed ([turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:7)). The user sees normal booking success, not a CAPTCHA error or degraded-mode warning.

3. User loads the page, waits a long time, then submits: there is no reliable explicit expiry message. In the common case, Cloudflare's default `refresh-expired="auto"` should refresh the token before submit, so the user sees no error. If a stale token is still submitted, the app returns only `Security check failed. Please try again.` ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:54), [BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:79)). If the hidden field is empty during refresh, the current fail-soft branch allows the submission through instead of surfacing expiry.

4. User submits the form multiple times rapidly: there is oversell protection, but not double-booking protection. The RPC locks the event row and re-checks capacity before insert, so concurrent requests should not exceed `total_capacity` ([20260313000000_event_bookings.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000000_event_bookings.sql:95)). But nothing stops the same user from creating multiple confirmed bookings while capacity remains. The UI disable is best-effort only, the server limiter is per IP and in-process only, and the database schema has no same-person uniqueness rule.

5. Bot submits without a Turnstile token: it is not blocked today. The booking action accepts an omitted token ([bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:27)), `verifyTurnstile()` returns `true` on `!token` ([turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:7)), and the existing booking action tests still pass on success paths with no `turnstileToken` in `VALID_INPUT` ([bookings.test.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/__tests__/bookings.test.ts:47), [bookings.test.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/__tests__/bookings.test.ts:94)). Practical effect: anonymous bots are gated only by the IP limiter.

6. The page is supposedly cached by ISR with `revalidate=60`: in repo reality, it is not. The production build emits `/l/[slug]` as `ƒ` dynamic server-rendered on demand, so middleware can generate a fresh CSP nonce per request ([middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:145)), the page can read that fresh nonce via `x-nonce` ([page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/page.tsx:130)), and `BookingForm` applies it to the external Turnstile `<Script>` ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:243)). No stale cached nonce issue was observed in the current build. The risk would reappear only if the route were made truly static later without redesigning nonce injection.

7. Very first render before `afterInteractive` fires: not literally before hydration, because the submit button is initially disabled while `firstName` and `mobile` state are empty ([BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:235)). But once React hydrates and the user fills the required fields, they can submit before the Turnstile script finishes loading because the button does not wait for widget readiness or token presence. In that window, the query returns no token and the current server-side behavior still allows the booking.
