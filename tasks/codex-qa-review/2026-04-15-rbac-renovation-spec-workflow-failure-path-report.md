**Workflow & Failure-Path Report**

**Findings**
- `Critical` (Scenarios 1, 7): The spec treats Phase 1 like a normal PR, but the DB rename is a deploy-order break. The migration changes persisted role strings immediately ([rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:94)), while `normalizeRole()` still accepts only legacy literals and `getCurrentUser()` returns `null` on anything else ([auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:18), [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:71)). Because the root layout calls `getCurrentUser()` on almost every page ([layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx:164)), a DB-first rollout or rollback to pre-compat code locks out the whole app. The spec needs a dual-vocabulary compatibility release first, DB migration second, cleanup last; rollback must target that compatibility build, not the old build.

- `High` (Scenario 2): The migration merges `reviewer` and `venue_manager` into `office_worker` ([rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:53)), but removes review decisions and event CRUD ([rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:31)). Today review queue access and decision writes are `reviewer`-specific ([reviews page](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/reviews/page.tsx:30), [events action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1323)), while venue managers still create/submit/edit drafts ([events new page](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/new/page.tsx:35), [events action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1150)). Without a queue-drain or reassignment plan, pending approvals and manager-owned drafts become stranded mid-workflow.

- `High` (Scenario 5): The verification strategy is too weak for the actual failure surface. `typecheck` will not catch Zod enums or raw role-string queries ([rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:131), [users action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:20), [reviewers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/reviewers.ts:14), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/users.ts:139)), and grepping migration history does not verify the live final schema or function bodies ([rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:108), [customers SQL](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260313000001_add_customers_and_consent.sql:49), [bookings RLS](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260410120002_tighten_event_bookings_rls.sql:13)). This needs post-migration inspection of live `pg_policies`/`pg_proc`/constraints plus role-matrix integration tests on a migrated database and action-layer tests for admin-client paths.

- `High` (Scenario 8): Planning is underspecified for `office_worker`. The spec grants create rights but only admin can “manage planning items” ([rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:38), [rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:151), [rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:183)), while current planning actions and RLS are uniformly `central_planner`-only ([planning action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/planning.ts:50), [planning action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/planning.ts:126), [planning RLS](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260225000001_tighten_planning_rls.sql:16), [planning RLS](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260225000001_tighten_planning_rls.sql:40)). The spec never says whether office workers can edit/delete their own items, tasks, or series, so the new workflow is either unusable or missing owner-based capability and RLS rules.

- `Medium` (Scenario 6): Debrief expansion is not phase-independent. The spec makes debrief create/read universal ([rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:35), [rbac spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-15-rbac-renovation-design.md:89)), but current page guard, action guard, and RLS all restrict to planner or creator-owned venue manager ([debrief page](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/debriefs/[eventId]/page.tsx:20), [debrief action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:50), [initial_mvp.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250218000000_initial_mvp.sql:262)). If RLS lands first, users still get UI/action denials; if actions/pages land first, saves fail at the DB boundary. Also the matrix lets `executive` create a debrief but never edit it afterward, which is not a coherent workflow.

- `Medium` (Scenario 3): `venue_id` handling is missing from the design even though current venue-scoped paths depend on it ([events new page](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/new/page.tsx:49), [bookings action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:192), [all-bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/all-bookings.ts:55), [customers.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/customers.ts:36)). If the migration clears `venue_id`, partial-rollout code breaks. If it keeps it, the spec must say whether `office_worker` still has venue-scoped behavior or whether it becomes metadata only, especially for ex-reviewers with `NULL` venue.

- `Advisory` (Scenario 4): JWT staleness is not the main breakage here. Both app auth and most RLS resolve role from `public.users` first ([auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:61), [current_user_role.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250301000000_secure_current_user_role.sql:8)), so a stale `central_planner` claim is usually harmless after the compatible code ships. The real risk is missed consumers that still hardcode old strings or consult JWT directly.

**Scenario Checks**
1. Migration ordering: unsafe as written; mitigate with a compatibility release that accepts both old and new role strings before the DB rename.
2. Partial migration: no, a former reviewer will not keep their assigned review workflow unless pending items are reassigned or temporary reviewer capability is retained; former venue managers also lose active draft/edit paths.
3. `venue_id` orphaning: do not drop or repurpose `venue_id` in Phase 1; keep data stable until every venue-scoped query is removed or explicitly remapped.
4. Session continuity: after compatible code ships, DB role wins over stale JWT claims; before that, new DB roles are treated as unauthenticated.
5. RLS gaps: verify live policies/functions/constraints after migration and run per-role access tests; grep/typecheck/build are insufficient.
6. Debrief expansion: deploy DB, page guards, and server actions together or behind a feature flag.
7. Rollback: there is no safe rollback to pre-migration code once the DB rename is live; either roll back to a compatibility build or reverse the DB migration first.
8. Planning permissions: the spec must decide whether `office_worker` can edit/delete its own planning items. If yes, add owner-based capability functions, action checks, and RLS; if no, call out the deliberate create-only limitation.

I did not run the app, tests, or migrations; this is a spec review against the current code and SQL paths.
