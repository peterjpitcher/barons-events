No Critical/High regressions found. The Turnstile migration is mostly fail-closed in production. The one material residual exposure is the added npm supply-chain dependency that now participates in loading the Cloudflare script.

1. `SECURE` — CSP bypass risk. The current CSP does include the fallback allowlist: `script-src 'self' 'nonce-…' 'strict-dynamic' https://challenges.cloudflare.com` and `frame-src https://challenges.cloudflare.com` in [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:60). The widget no longer passes a nonce because [turnstile-widget.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/turnstile-widget.tsx:39) sets `injectScript` with only `scriptOptions={{ appendTo: "body" }}`, even though the installed library supports `scriptOptions.nonce` and defaults `injectScript` to true in [index.d.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/node_modules/@marsidev/react-turnstile/dist/index.d.ts:362). Under Next’s nonce model, framework/page bundles are nonced automatically, so a trusted Next script can inject Turnstile under `strict-dynamic`; the explicit `https://challenges.cloudflare.com` host source is the legacy fallback. This is not a current CSP bypass or violation, but passing the nonce would be tighter and closer to Cloudflare’s preferred setup.

2. `SECURE` — Timing attack. [verifyTurnstile](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:10) does not compare any secret locally. The only direct comparison is `data.action !== action` at [turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:42), and `action` is not secret. Response time differs between “missing token”, “missing secret”, and “siteverify call happened”, but that does not leak credentials or the Turnstile secret.

3. `SECURE` — Token replay. I found no app-side token tracking beyond [verifyTurnstile](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:30), but every protected path calls `siteverify` before doing work: [auth login](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:88), [password reset](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:242), and [booking](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:49). Cloudflare documents tokens as single-use with `timeout-or-duplicate` on replay, so server-side replay protection is delegated correctly to Cloudflare.

4. `SECURE` — CAPTCHA bypass when the widget renders `null`. If `NEXT_PUBLIC_TURNSTILE_SITE_KEY` is absent, [TurnstileWidget](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/turnstile-widget.tsx:26) returns `null`. Auth actions then read a missing `cf-turnstile-response` and strict-mode reject it at [turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:15); booking reads the hidden field manually and falls back to `""` at [BookingForm.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx:54), which is also rejected in strict mode by [bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:50). Injecting a fake `cf-turnstile-response` field does not bypass anything in production because the token is still verified against Cloudflare. Caveat: in non-production only, a non-empty fake token will pass if `TURNSTILE_SECRET_KEY` is unset, due to the explicit dev convenience branch at [turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:21).

5. `RISK Low` — XSS via third-party script / supply chain. This is not an app-side XSS sink, but the migration does add a new trust boundary: `@marsidev/react-turnstile` in [package.json](/Users/peterpitcher/Cursor/BARONS-BaronsHub/package.json:18), locked to `1.5.0` with integrity in [package-lock.json](/Users/peterpitcher/Cursor/BARONS-BaronsHub/package-lock.json:1482). The installed package hardcodes Cloudflare’s Turnstile script URL in [index.d.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/node_modules/@marsidev/react-turnstile/dist/index.d.ts:436) and auto-injects it. If either the npm package or Cloudflare-hosted script were compromised, that script would execute with full origin privileges. That risk is inherent to the new wrapper-based loading path, though likelihood is low and lockfile integrity reduces tampering risk.

6. `SECURE` — Form data leakage. The library’s default behavior is to add only the hidden response field `cf-turnstile-response` as documented in [index.d.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/node_modules/@marsidev/react-turnstile/dist/index.d.ts:173), and the app only reads that token. Cloudflare’s widget docs describe the hidden token field being submitted with the form to your server, not arbitrary form values being sent to Cloudflare. Cloudflare still receives Turnstile signals such as IP, TLS fingerprint, user-agent, sitekey, and origin as part of the challenge service, but I found no evidence in the package or docs that email/password/booking fields are exfiltrated to Cloudflare.

7. `SECURE` — Dead code in `src/app/l/[slug]/page.tsx`. The nonce read at [page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/page.tsx:133) is unused and has no direct security effect. The actual nonce-bearing script path lives in the root layout at [layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx:155) and [layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx:164), so this page-level read is just cleanup debt, not a vulnerability.

Outside your checklist, the main hardening gap is [turnstile.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts:42): it checks `action` but not `hostname`, and it does not send `remoteip`. Cloudflare’s current guidance recommends validating both `action` and `hostname` when specified. If this sitekey is valid on more than one hostname, adding hostname validation would tighten abuse resistance.
