**Spec Defects**
- `office_worker` scoping is undefined. Current `venue_manager` logic is venue-bound, while `reviewer` logic is assignee-bound; the spec merges them without saying whether `venue_id` still authorizes anything. [events lib](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:173), [customers lib](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/customers.ts:37)
- The rollout plan ignores JWT/session mismatch. `getCurrentUser()` hard-whitelists role strings, while `current_user_role()` can still fall back to JWT role data; bulk DB migration without deploy ordering can lock users out or create inconsistent auth decisions. [auth](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:18), [current_user_role sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250301000000_secure_current_user_role.sql:1)
- `src/actions/users.ts` hardcodes four-role Zod enums, but the spec does not call that out. [users action](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:20)
- Reviewer-specific logic is omitted: `src/lib/reviewers.ts`, venue `default_reviewer_id`, automatic assignee fallback, and the review queue all assume reviewer identity still exists. [reviewers](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/reviewers.ts:14), [events assignee](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:1244)
- Notification logic cannot stay “out of scope”: planner recipient lookups are hardcoded in `notifications.ts`. [notifications](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:629)
- The `typecheck` mitigation is overstated because generated Supabase types keep `users.role` as `string`. [supabase types](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/types.ts:17)
- The SQL scope is understated: the event import migration and planning-calendar seed migration both reference `central_planner`. [import sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260206120000_import_baronspubs_2026_events.sql:23), [planning seed](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120007_seed_planning_calendar_2026_2027.sql:13)
- The scope estimate is materially low. I counted 47 non-test `src` files with legacy role literals, not “~23 pages”; only 17 page routes in this checkout contain those literals.

**Unresolved Ambiguities**
- Executive is described as a read-only observer, but the matrix grants executive `Create/read debriefs`. Those two statements conflict.
- The spec never defines the exact read scope for `office_worker` on events, bookings, customers, or artists.
- It is unclear whether `office_worker` participates in review assignment at all, or whether approval becomes fully administrator-only with no assignees.
- “Read-only event details from planning” does not say whether SOP task toggles, assignee controls, or any other embedded server actions remain visible.
- “Single migration file” and “inverse migration” are underspecified: it is not clear whether rollback is required in the same PR or whether historical seed/import SQL must also be rewritten.
