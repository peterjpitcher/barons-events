# Workflow & Failure-Path Report v2

## Summary
v2 materially improves the v1 design, especially by moving multi-row event creation into an RPC, adding parent-row locking for cascade completion, adding `upload_status`/`uploaded_at`, and naming explicit status-consumer sweeps.

The remaining production risks are mostly around ambiguous outcomes and rollout boundaries: HTTP/RPC timeouts without idempotency, queue jobs without claim/retry/dead-letter semantics, attachment confirm failure states, service-role behaviour inside `SECURITY DEFINER` triggers, stale `approved_pending_details` proposals, and merge-order gaps where migrations can land before consumers.

## New Partial Failure Findings (WF-V2-NNN)
- **WF-V2-001 — Multi-venue RPC timeout remains ambiguous.** If the HTTP request times out, Postgres may still commit or may cancel; v2 has no `client_request_id`, `multi_venue_batch_id` uniqueness, or result lookup. The audit batch ID is metadata only ([spec:476](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:476)). Retrying can duplicate events unless the RPC is idempotent.

- **WF-V2-002 — RPC atomicity depends on exception propagation not stated in implementation detail.** Existing SOP SQL is atomic per checklist ([sop RPC:26](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120003_add_sop_rpc_functions.sql:26), [sop RPC:233](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120003_add_sop_rpc_functions.sql:233)), but current callers catch SOP errors ([events.ts:887](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:887), [planning.ts:95](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/planning.ts:95)). The new RPC must not catch per-event SOP failures.

- **WF-V2-003 — Backfill queue has no claim, retry, or dead-letter model.** The table has `processed_at` and `error` only ([spec:823](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:823), [spec:828](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:828)). It needs `attempt_count`, `last_attempt_at`, `locked_at/locked_by`, `next_attempt_at`, and a dead-letter threshold.

- **WF-V2-004 — Queue processing atomicity is undefined.** v2 does not say whether child inserts and `processed_at` update happen in one DB transaction. Existing cron routes loop in application code and catch per item ([sms-reminders:39](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/sms-reminders/route.ts:39), [sms-reminders:53](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/sms-reminders/route.ts:53)); that pattern would be unsafe for `pending_cascade_backfill`.

- **WF-V2-005 — Attachment confirm can mark the wrong row unless ownership is rechecked.** Confirm parses only `attachmentId` in the spec ([spec:1061](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1061)). If implemented with service-role updates, it must verify `uploaded_by = current user` or parent edit permission before reading Storage and updating.

- **WF-V2-006 — Failed attachment rows linger indefinitely.** Cleanup only selects old `pending` rows and soft-deleted rows ([spec:1093](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1093), [spec:1097](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1097)). Rows set to `upload_status = 'failed'` by confirm are not cleaned up.

- **WF-V2-007 — Attachment object key derivation is internally inconsistent.** The object key uses a `safe_extension` derived from sniffed MIME ([spec:897](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:897)), but `requestAttachmentUploadAction` must return `storagePath` before upload and before sniffing ([spec:1058](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1058), [spec:1060](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1060)).

- **WF-V2-008 — Labour rate snapshot is not atomic unless implemented as one SQL statement/RPC.** v2 says the action re-reads then inserts ([spec:244](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:244), [spec:251](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:251)). Current debrief persistence is a separate upsert call ([debriefs.ts:111](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:111), [lib/debriefs.ts:27](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/debriefs.ts:27)). A rate change between read and upsert creates an arguable stale snapshot.

- **WF-V2-009 — Stale `approved_pending_details` has no terminal path.** v2 only blocks approval after `start_at` has passed ([spec:618](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:618)). Once approved, a venue manager can abandon the full form and the row stays `approved_pending_details`.

- **WF-V2-010 — Event status trigger may block legitimate server-action transitions.** The trigger checks `public.current_user_role() != 'administrator'` ([spec:564](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:564), [spec:567](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:567)). `current_user_role()` reads `auth.uid()` / JWT claims, not the TypeScript `user` object ([user_deactivation.sql:128](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260416000000_user_deactivation.sql:128), [user_deactivation.sql:143](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260416000000_user_deactivation.sql:143)). Server actions using service-role clients, like current debrief status updates, will not look like `administrator` to this trigger ([debriefs.ts:132](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:132)).

- **WF-V2-011 — SLT BCC privacy is false without a shared alias.** v2 says use a shared alias if configured, else the first recipient as `to` and the rest as `bcc` ([spec:319](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:319)). With 10 SLT members and no alias, the first recipient’s address is visible in the `To` header.

- **WF-V2-012 — Empty SLT list audit is contradictory.** Empty list means no email sent ([spec:317](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:317), [spec:348](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:348)), but the debrief success audit always says `meta.slt_emailed: true` ([spec:338](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:338)).

- **WF-V2-013 — Proof-read menus can duplicate manual data.** `sop_task_templates` has no unique title constraint ([sop tables:71](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120000_add_sop_tables.sql:71), [sop tables:82](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120000_add_sop_tables.sql:82)). `ON CONFLICT (id)` only protects reruns, not an existing “Proof-read menus” row with a different UUID.

- **WF-V2-014 — Category-change backfill has no concrete queue contract.** v2 mentions `updateVenueAction` for category changes into a matching filter ([spec:815](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:815)), but the queue row only contains `venue_id` ([spec:823](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:823), [spec:825](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:825)). It cannot record old/new category or dedupe repeated category flips.

## New Race Conditions
- **Parent-row lock path is sound for two child completions.** If child A gets the parent lock first and child B has already updated but is blocked in its trigger, A sees B’s old open status and leaves the parent open. After A commits, B reads parent `open`, sees no open siblings, and completes the parent. If A completed the parent first, B later sees parent `done` and the `v_parent_status = 'open'` guard prevents a second update ([spec:722](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:722), [spec:741](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:741)).

- **Overlapping cron invocations can process the same backfill row.** v2 says the cron runs every minute ([spec:820](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:820)) but has no `FOR UPDATE SKIP LOCKED`, claim token, or `locked_at` column.

- **Debrief rate read/write can interleave with admin settings updates.** Use one SQL statement or RPC that inserts `labour_rate_gbp_at_submit` from `business_settings` at write time; do not do read-then-upsert in application code ([spec:251](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:251), [lib/debriefs.ts:29](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/debriefs.ts:29)).

- **Direct task metadata updates bypass cascade reopen.** The cascade trigger is `after update of status` ([spec:772](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:772)) and returns on no status change ([spec:718](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:718)). An update that rewrites `completed_at` on an open child will not repair parent or child invariants.

## New Retry / Idempotency Gaps
- **Multi-venue create needs idempotency.** `multi_venue_batch_id` should be persisted on created rows or in a batch table with a unique client key, not only audit metadata ([spec:476](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:476)).

- **Queue retry needs row-level status.** `processed_at` should only be set after all child inserts and audit rows commit. Failed rows need retryable `error`, attempts, and dead-letter handling ([spec:827](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:827), [spec:828](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:828)).

- **Attachment confirm should distinguish transient and terminal failures.** Storage object not yet present or download failure should keep the row `pending` and be retryable; MIME mismatch or unidentifiable type should be terminal `failed` ([spec:1063](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1063), [spec:1066](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1066)).

- **SLT email failure still has no retry.** v2 audits delivery failure but does not queue retry ([spec:320](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:320)). Existing notification helpers swallow send failures after logging ([notifications.ts:702](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:702), [notifications.ts:704](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:704)).

## New Stale / Cached Data
- **`approved_pending_details` can become stale after event start.** No reaper, archive, or revert-to-draft path is defined after approval ([spec:520](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:520), [spec:621](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:621)).

A pub-to-cafe change can leave pub-only children blocking a master unless they are marked out-of-scope or `not_required`.

- **Planning specific-venue selections are form state only.** v2 explicitly says client state ([spec:486](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:486)); current planning editor also keeps venue selection in component state ([planning-item-editor.tsx:52](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-item-editor.tsx:52), [planning-item-editor.tsx:67](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-item-editor.tsx:67)). ## New Empty / Edge States
- **Per-venue SOP template with no eligible managers can create a master with no children.** v2 skips venues with missing/deactivated default managers ([spec:701](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:701), [spec:704](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:704)). If every venue is skipped, the master never auto-completes.

- **Empty SLT list should audit `slt_emailed: false`.** Current v2 wording says no email but success audit says true ([spec:338](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:338), [spec:348](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:348)).

- **`file-type` null result is not defined.** Treat it as terminal failed for allow-listed binary types unless supplementary OOXML checks pass ([spec:1064](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1064), [spec:1102](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1102)).

- **Failed/pending attachments are not user-visible under RLS.** The read policy only exposes `upload_status = 'uploaded'` rows ([spec:942](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:942), [spec:945](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:945)). ## New User-Flow Breakpoints
- **User retry after RPC timeout can duplicate events.** The success banner is clear only when the action returns ([spec:470](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:470)); it does not cover “request failed but DB committed”.

- **Stale approved proposal has no user-facing resolution.** Venue managers can be asked to complete details after the event date has passed, because only approval-time validation is specified ([spec:587](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:587), [spec:620](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:620)).

- **Specific-venues mode does not survive reload.** If that is acceptable, state it. If not, persist selected venue IDs in URL/search params or draft storage ([spec:486](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:486)).
