**Security & Data Risk Report**

**Findings**
- `SEC-001 High` Removing idle timeout materially weakens protection for unattended browsers. The authenticated layout no longer mounts any idle-sign-out logic, the custom session cookie is explicitly persistent for 24 hours, and server validation now ignores inactivity and expires only on `expires_at`. On an unlocked or shared machine, a walk-up user can keep using the prior user’s session for the remaining lifetime. Refs: [src/app/layout.tsx#L157](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx#L157), [src/app/layout.tsx#L168](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx#L168), [src/lib/auth/session.ts#L22](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L22), [src/lib/auth/session.ts#L27](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L27), [src/lib/auth/session.ts#L95](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L95).
- `SEC-002 Low` `last_activity_at` is now dead security metadata, but several code paths still imply idle-expiry exists. It is still written and returned on validation, the audit event union still includes idle expiry, the login page still renders an inactivity message, and the cron comment still claims idle cleanup. That creates a false sense of security for operators, auditors, and future maintainers. Refs: [src/lib/auth/session.ts#L62](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L62), [src/lib/auth/session.ts#L111](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L111), [src/lib/audit-log.ts#L53](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/audit-log.ts#L53), [src/app/login/page.tsx#L62](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx#L62), [src/app/api/cron/cleanup-auth/route.ts#L8](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/cleanup-auth/route.ts#L8).

**Assessment**
- `Security impact:` The removal is a real regression in session-hardening, but I did not find a new code-path auth bypass on protected page requests. The main change is exposure window: 30 minutes idle became 24 hours absolute-only.
- `Shared/public computers:` Risk is materially higher. If a user leaves a machine unlocked, the next person can continue in that account until expiry. The custom cookie itself survives browser restart for 24 hours; full post-restart access also depends on Supabase cookie persistence.
- `Heartbeat endpoint:` Removal is net-positive. I found no live `/api/auth/heartbeat` route, and the current `/api` surface is cron or API-key protected rather than session-renewal based. Refs: [src/app/api/cron/cleanup-auth/route.ts#L12](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/cleanup-auth/route.ts#L12), [src/app/api/v1/events/route.ts#L30](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/events/route.ts#L30), [src/app/api/v1/venues/route.ts#L13](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/v1/venues/route.ts#L13).
- `Absolute expiry:` Still enforced correctly on protected requests. Middleware validates the Supabase JWT, requires `app-session-id`, calls `validateSession()`, clears the cookie on invalid/expired sessions, and still checks `session.userId === user.id`. Refs: [middleware.ts#L196](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L196), [middleware.ts#L214](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L214), [middleware.ts#L228](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L228), [middleware.ts#L236](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L236), [middleware.ts#L244](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L244), [src/lib/auth/session.ts#L95](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L95).
- `Cron cleanup:` Still correct for the new model. `cleanupExpiredSessions()` deletes rows where `expires_at < now`, and the cron route remains bearer-protected by `CRON_SECRET`. Refs: [src/lib/auth/session.ts#L144](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L144), [src/lib/auth/session.ts#L148](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L148), [src/app/api/cron/cleanup-auth/route.ts#L16](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/cleanup-auth/route.ts#L16).
- `Auth bypass introduced by removal:` None found in the live route set. Current protected page flow still fail-closes. I also found no live session-authenticated `/api` handlers depending on the removed idle logic.
- `Session fixation protections:` Still intact. Session IDs are freshly generated on creation, and middleware still rejects any app session whose `userId` does not match the validated Supabase user. Refs: [src/lib/auth/session.ts#L41](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L41), [middleware.ts#L244](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L244).

**Open Questions**
- `Re-auth flow after absolute expiry:` middleware redirects expired custom sessions to `/login`, but the login page redirects any still-valid Supabase user away immediately. If Supabase outlives the custom session, this looks prone to a redirect loop rather than a clean re-login. That is not an auth bypass, but it is worth testing. Refs: [middleware.ts#L230](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L230), [src/app/login/page.tsx#L41](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx#L41).
- `Policy decision:` If kiosk/shared-office use is in scope, 24-hour absolute-only sessions are weak without compensating controls such as shorter expiry, session cookies without `maxAge`, or step-up re-auth for sensitive actions.
