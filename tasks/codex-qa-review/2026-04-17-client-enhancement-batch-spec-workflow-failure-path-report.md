# Workflow & Failure-Path Report: Client Enhancement Batch Spec

## Summary
I found production-grade failure paths in the spec around batch atomicity, SOP side effects, cascade races, stale approval states, direct uploads, and audit rollout. The recurring pattern is: the spec asks for multi-row workflows but does not define a transactional boundary, idempotency key, partial-result contract, or recovery UI.

## Partial Failure Findings (WF-NNN ids)
- **WF-001 — Multi-venue event creation can leave a half-created batch.**  
  Wave 3.3 requires one `events` row per selected venue and a success banner saying “Created 5 events” [spec:385](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:385), [spec:392](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:392). Current creation is a single Supabase insert per event, not a transaction [src/lib/events.ts:465](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:465). If venue 3 of 5 fails in a loop, venues 1-2 remain committed and the existing action shape returns only “Could not save the draft” [src/actions/events.ts:952](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:952). A user retry then creates duplicates for the first two venues. The spec needs either one RPC transaction for the whole batch, or a batch result that says exactly which venue rows were created and which are safe to retry.

- **WF-002 — Event SOP fan-out can silently fail after event insert.**  
  The spec says each event “auto-generates its own SOP checklist” [spec:391](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:391), but current event save catches SOP/planning bridge failure and continues [src/actions/events.ts:878](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:878). The helper inserts the planning item before calling SOP generation [src/lib/events.ts:553](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:553), [src/lib/events.ts:569](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:569). Result: event exists, possibly a planning item exists, but no SOP tasks; the UI still reports draft creation success. This is an operational miss, not just a log warning.

- **WF-003 — Cascade expansion has the same mid-loop partial failure.**  
  Wave 5.2 inserts one child task per matching venue and audits each child [spec:556](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:556), [spec:569](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:569). Existing task creation is one insert [src/lib/planning/index.ts:841](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:841), [src/lib/planning/index.ts:855](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:855). If child 3 fails, the master and children 1-2 persist, and there is no specified retry contract or uniqueness constraint such as `(parent_task_id, cascade_venue_id)` to make replay safe.

- **WF-004 — New-venue cascade back-fill runs inside the venue save request with unbounded work.**  
  The spec puts back-fill inside `createVenueAction` / `updateVenueAction`, scanning all open masters and spawning children [spec:612](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:612), [spec:615](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:615). Current venue creation is synchronous and returns immediately after insert/audit/revalidate [src/actions/venues.ts:56](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/venues.ts:56). With many open masters, adding one pub can become a slow or timing-out admin request, with unclear partial back-fill state if the hook fails mid-way.

- **WF-005 — Attachment upload state cannot work as specified because `uploaded_at` is missing.**  
  The attachments table has `created_at` and `deleted_at`, but no `uploaded_at` [spec:672](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:672). The server action then says confirm “flips `uploaded_at`” [spec:704](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:704), and the orphan sweep queries `uploaded_at IS NULL` [spec:749](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:749). That migration will either fail at implementation or ship uploads that cannot be confirmed or cleaned up.

- **WF-006 — Proof-read menus migration will fail on `ON CONFLICT`.**  
  The spec requires `ON CONFLICT DO NOTHING` on `(section_id, title)` [spec:133](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:133). Current `sop_task_templates` has no unique constraint on `(section_id, title)`; it only defines the columns and non-unique indexes [add_sop_tables.sql:71](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120000_add_sop_tables.sql:71), [add_sop_tables.sql:82](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120000_add_sop_tables.sql:82). PostgreSQL will reject that `ON CONFLICT` target.

## Race Conditions
- **WF-007 — Cascade parent auto-complete is still race-prone.**  
  The spec says a DB trigger “eliminates races if two children finish at once” [spec:581](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:581). The trigger only queries for open siblings and then updates the parent [spec:589](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:589). Two separate transactions can update two different children at the same time; each trigger can still see the other sibling as open, so neither updates the parent. Double-updating the parent is mostly contained by `and status not in (...)` [spec:596](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:596), but the real production failure is a parent left open after all children are done. The trigger needs a parent-row lock, advisory lock, or a statement-level reconciliation job.

- **WF-008 — Cascade reopen is promised but not implemented in the trigger.**  
  The spec states “if a child reopens ... reopen the parent too” [spec:608](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:608). The trigger exits immediately unless `new.status` is `done` or `not_required` [spec:587](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:587). Reopening a child will therefore leave the parent marked done, hiding unfinished work.

## Retry / Idempotency Gaps
- **WF-009 — Direct upload confirm/delete races are undefined.**  
  The spec recommends presigning, direct client upload, then `confirmAttachmentUploadAction(attachment_id)` [spec:699](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:699). It does not define confirm semantics when the client loses connection mid-upload, never confirms, or the row is soft-deleted before confirm [spec:705](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:705). That creates stuck pending rows or orphaned storage objects. Two simultaneous uploads with the same filename are sound because the storage path includes `attachment_id` and is unique [spec:667](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:667), [spec:676](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:676).

- **WF-010 — Audit guard rollout will block CI if implemented literally.**  
  The spec says the guard checks for `logAuditEvent` [spec:315](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:315), but the repo uses `recordAuditLogEntry()` and `logAuthEvent()`, not `logAuditEvent()` [src/lib/audit-log.ts:32](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/audit-log.ts:32), [src/lib/audit-log.ts:99](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/audit-log.ts:99). It also has several mutating functions still unaudited, for example `togglePlanningTaskStatusAction` updates status without audit [src/actions/planning.ts:572](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/planning.ts:572). The guard must come after coverage or ship with an allowlist generated from the gap map.

## Stale / Cached Data
- **WF-011 — Pending approvals can go stale past `start_at`.**  
  Pre-event submission only validates `start_at` as future at creation [spec:490](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:490). There is no reaper, expiry state, or approval-time revalidation in the spec [spec:485](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:485). A venue manager can submit an event, wait a week, and the admin can still see or approve a now-past proposal unless extra rules are added.

- **WF-012 — Existing event status surfaces will mis-handle `pending_approval`.**  
  The new status is added in the spec [spec:445](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:445), but current `EventStatus` excludes it [src/lib/types.ts:43](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/types.ts:43), status counts exclude it [src/lib/events.ts:761](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:761), and detail copy falls back to draft for unknown statuses [src/app/events/[eventId]/page.tsx:31](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/[eventId]/page.tsx:31), [src/app/events/[eventId]/page.tsx:80](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/[eventId]/page.tsx:80). Pending rows will be undercounted or shown with the wrong affordances unless the spec requires a full status-surface sweep.

- **WF-013 — Venue category changes can leave cascades waiting on irrelevant children.**  
  The spec back-fills when a venue is created or changes category into a matching filter [spec:612](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:612), and later says pub→cafe children stay as ordinary tasks [spec:792](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:792). That means a pub-only master can remain blocked by a child for a venue that is now a cafe. If that is intended, the UI must mark the child as out-of-scope and tell the user; otherwise the parent completion rule becomes stale.

## Empty / Edge States
- **WF-014 — Labour hours validation permits operationally impossible values and has a schema mismatch.**  
  The spec uses `numeric(6,2)` for `labour_hours` [spec:157](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:157), but Zod allows up to `99999.99` [spec:204](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:204). `2500` hours would be accepted and materially distort labour cost reporting; `25000` would pass Zod but fail the database precision. Add a realistic hard cap, warning threshold, and align DB precision with validation.

- **WF-015 — `not_required` still has done-only candidate views.**  
  The spec correctly calls for a sweep [spec:117](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:117). Candidate files that will show wrong state or counts if missed: [sop-checklist-view.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/sop-checklist-view.tsx:49), [planning-task-list.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-task-list.tsx:133), [unified-todo-list.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/todos/unified-todo-list.tsx:165), and the generic update path in [src/lib/planning/index.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:883). `PlanningItemCard` is already sound for resolved counts [planning-item-card.tsx:59](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-item-card.tsx:59).

## User-Flow Breakpoints
- **WF-016 — Multi-venue planning mode switching is under-specified.**  
  The spec introduces “Global” versus “Specific venues” and joint validation [spec:408](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:408), [spec:413](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:413). Current planning UI keeps venue as a single controlled field with `Global` as an empty value [planning-item-editor.tsx:67](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-item-editor.tsx:67), [planning-item-editor.tsx:273](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-item-editor.tsx:273). With a new multi-select, flipping modes mid-form can either preserve hidden venue selections and create venue-specific rows when the user thinks they chose global, or clear them and lose work. The server contract should say `mode=global` ignores `venue_ids`, and the client should preserve selections when toggling back to “Specific venues”.

- **WF-017 — SLT debrief email failures are not discoverable.**  
  The spec says Resend errors are logged and not thrown [spec:263](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:263), and calls delivery status on the detail page optional v2 [spec:296](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:296). Existing notification helpers swallow send failures with `console.warn` [src/lib/notifications.ts:695](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:695), [src/lib/notifications.ts:703](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:703), while debrief submission still returns success [src/actions/debriefs.ts:195](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:195), [src/actions/debriefs.ts:199](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/debriefs.ts:199). A Resend 429 or outage loses the SLT notification with no retry queue and no admin-visible failure.
