# Assumption Breaker Report v5

## Summary

v5 is **not quite ready to implement**. Most v4 SQL-level blockers are fixed, especially the multi-venue RPC authorisation blocks, audit CHECK, migration order, trigger grants, and draft return shape.

The remaining blocker is Wave 3.3: `preApproveEventAction` now says it creates the planning item and SOP, but the planning item insert is under-specified and the “single transaction including email” wording is not implementable through the current server-action pattern. There are also two stale prose/changelog contradictions that should be cleaned before implementation.

Verdict: **NEEDS V6**.

## PART B — New v5 Issues (AB-V5-NNN)

**AB-V5-001 — BLOCKER: `preApproveEventAction` planning item insert is still under-specified.**  
Wave 3.3 says approval creates the linked `planning_items` row using venue, target date, title, and placeholder `type_label`: [spec:948](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:948). It does not explicitly set `status = 'planned'`, but `planning_items.status` is `not null` and checked against the allowed set: [planning schema:42](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260223120000_add_planning_workspace.sql:42). It also only implies `event_id`; the event link column and one-item-per-event index are explicit in the schema: [event link migration:5](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120002_add_event_planning_link.sql:5), [event link migration:9](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260408120002_add_event_planning_link.sql:9). Existing event planning creation sets `event_id`, `title`, `type_label`, `venue_id`, `target_date`, `status`, and `created_by`: [events.ts:553](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:553), [events.ts:563](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:563). Current migrations later relax `planning_items.created_by` nullability: [user deactivation migration:36](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260416000000_user_deactivation.sql:36), but the spec should still state the intended actor.

Required v6 wording: insert `planning_items(event_id, title, type_label, venue_id, target_date, status, created_by)` with `event_id = event.id`, `type_label = 'Event'` or another named placeholder, `status = 'planned'`, and `created_by = admin_id` or the original creator, whichever is intended.

**AB-V5-002 — BLOCKER: “single transaction” for `preApproveEventAction` is asserted but not implementable as a server action.**  
The spec says status update, planning item insert, SOP generation, audit, and email all happen in a single transaction and any failure rolls back: [spec:948](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:948). The existing pattern is multiple Supabase requests: insert planning item, then call SOP RPC: [events.ts:553](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:553), [events.ts:569](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:569). Existing event creation even catches and suppresses SOP failures: [events action:878](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:878), [events action:889](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/events.ts:889). Email also cannot literally roll back with DB state; the current review-decision email helper catches and logs failures: [notifications.ts:344](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:344), [notifications.ts:375](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:375).

Required v6 design: define a dedicated direct-call RPC, e.g. `pre_approve_event_proposal(event_id, admin_id)`, that does the DB work atomically: status update, planning item insert, SOP generation, audit, optional notification-outbox insert. Send email after commit or via outbox. Do not describe external email delivery as part of the rollbackable DB transaction.

**AB-V5-003 — MAJOR: stale authorisation prose still describes the old no-venue office-worker allowance.**  
The SQL is corrected, but behaviour step 4 still says office workers pass when `u.venue_id IS NULL OR u.venue_id = venue_id`: [spec:552](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:552). That is the exact bad shape v4 was meant to remove. Wave 3 permissions also use the shorthand `canManageEvents(role, venueId)` for every venue: [spec:1002](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1002), while the helper only checks “office worker has a venue”, not “office worker matches this target venue”: [roles.ts:20](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/roles.ts:20).

Required v6 wording: administrator OR office worker with `user.venue_id IS NOT NULL AND user.venue_id = target venue_id`.

The operative convention is correct: trigger functions get no service-role grant: [spec:66](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:66). **NEEDS V6.**

Minimum v6 changes:

4. Remove or correct the stale SECURITY DEFINER changelog line that says every function grants `service_role`.
