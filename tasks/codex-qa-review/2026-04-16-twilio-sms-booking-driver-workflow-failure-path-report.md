Reviewed against the spec and repo reality. The biggest design risks are retry state, inbound idempotency, ambiguous reply routing, and compliance around opt-out/stale numbers.

| # | What Happens | Severity | Likelihood | Recommended Mitigation |
|---|---|---:|---:|---|
| 1 | Twilio failure after claim creates `sms_campaign_sends` row. Next cron excludes that wave because the row exists, even though no SMS was sent. Transient Twilio outages become permanent skips. | High, critical during outage | Medium | Add campaign send state: `claimed`, `sent`, `failed`, `permanent_failed`, `attempt_count`, `last_error`, `next_retry_at`. Only suppress future sends for successful `sent` rows. Mark invalid numbers as permanent failure; retry 429/5xx/network errors with backoff. Store `sent_at` only after Twilio succeeds, or separate `claimed_at` from `sent_at`. |
| 2 | If all campaign rows are `converted_at != null`, inbound lookup finds no active campaign and sends the generic “not sure which event” fallback. A delayed duplicate after a successful booking may confuse the customer. | Medium | Medium | Keep recent converted campaign context with `converted_booking_id` and reply “You’re already booked for X” instead of generic fallback. Add `expires_at` and explicit states so expired, converted, and unknown replies produce different messages. Store inbound `MessageSid` for duplicate-safe responses. |
| 3 | The booking RPC enforces current capacity, so the reply returns sold out if capacity is gone. However, later waves can still send for a sold-out event because the campaign audience/cron spec does not skip zero-capacity events. | Medium-High | Medium | Recompute capacity before every send and skip capped events with no remaining capacity. Also recheck capacity immediately before claim/send. If capacity later increases, allow future waves again. Consider a specific waitlist/unavailable response rather than generic promo copy when capacity is tight. |
| 4 | With the spec’s intended UK calendar-day math, an 08:30 UTC same-day event should not fire at 08:00 UTC; wave 3 should have fired the previous UK calendar day. If implementation uses exact-hour `ceil` math, it may incorrectly send 30 minutes before the event. If yesterday’s run was missed, there is no catch-up behavior. | Medium | Medium | Implement date diff with `Europe/London` calendar dates, not millisecond offsets. Add tests for early UTC events and DST. Define catch-up rules explicitly: likely “never send same-day promo SMS,” and log/alert missed waves instead of sending late. |
| 5 | The webhook picks the most recent active campaign. If Event B was sent after Event A, reply `3` books Event B even if the customer meant Event A. | High | Medium | Do not use plain numeric replies when multiple active campaigns exist. Add a campaign/event reply token, e.g. “Reply `A 3`”, or disable reply-to-book and use links when a customer has more than one active campaign. Alternative: two-step confirmation, but a token is more reliable. |
| 6 | A single cron processing 100K SMS will likely hit Twilio throughput limits and/or Vercel execution limits. Claim-before-send rows created before 429/timeouts may suppress retries permanently. Partial completion is not resumable. | Critical | Medium now, high at scale | Move to a durable queue/job table with batch limits, send states, `FOR UPDATE SKIP LOCKED`, retry/backoff, and max sends per invocation. Respect Twilio 429/`Retry-After`. Process in chunks and make the cron resumable. Configure Twilio Messaging Service throughput before scaling. |
| 7 | Cancellation does not clear `converted_at`, so the customer will not receive wave 3 after cancelling. That may be acceptable for voluntary cancellations, but it conflicts with “cancellation reverses the booking” if the business expects re-promotion. | Low-Medium | Medium | Define product policy. If cancellations should reopen eligibility, store `conversion_booking_id` and `conversion_cancelled_at`, and only re-include selected cancellation types. For voluntary cancellation, keeping suppression is probably safer than re-marketing immediately. |
| 8 | Cron fetch excludes `deleted_at IS NULL`, so wave 2 should not fire after soft delete. But existing active campaign rows remain. A reply after deletion can still select the deleted event, then `create_booking` returns `not_found`; the spec has no explicit customer-facing message for that case. | Medium-High | Low-Medium | On inbound, join campaign to event and require `deleted_at IS NULL`, bookable status, `booking_enabled`, and future `start_at`. Mark open campaign rows as `cancelled`/`event_unavailable` when an event is soft-deleted. Return “This event is no longer available” rather than a generic fallback. |
| 9 | If the website booking happens after the audience was fetched or after the SMS was sent, the customer can still receive wave 2. Suppression at 08:03 cannot prevent an SMS already sent at 08:01. | Medium | Medium | Do just-in-time suppression in the claim SQL: insert/send only if no confirmed booking exists at claim time and customer is still eligible. Ideally set `customer_id` and campaign suppression inside the booking transaction/RPC, or via a DB trigger, so suppression is immediate. |
| 10 | Two identical inbound replies can both see the same active campaign before `converted_at` is set. The current `create_booking` RPC is not idempotent; both requests can create bookings until capacity/per-mobile caps stop them. | Critical | Medium | Add inbound idempotency using Twilio `MessageSid` with a unique `sms_inbound_messages` table and stored response. Also create a transactional `create_booking_from_campaign` RPC that locks the campaign/customer/event, creates one booking, marks converted, and rejects duplicates. Consider a unique `source_campaign_id` or `inbound_message_sid` on bookings. |
| 11 | Campaign sends to the stale number in `customers.mobile`. If the number is recycled, another person could receive the promo and reply; lookup by mobile may book under the old customer’s identity. | High | Low-Medium | Track phone verification and deliverability: `mobile_verified_at`, `last_seen_at`, `invalidated_at`, `marketing_opt_in_at`, and failure status. Store `sent_to_mobile` on campaign rows. Stop sending to numbers with Twilio permanent failures or stale consent. When customers update numbers, revoke/suppress the old number. |
| 12 | Wave 2 audience query respects `marketing_opt_in = true`, so revoked opt-in is honored if it happens before audience selection. If opt-out happens after audience fetch/claim but before send, the SMS can still go out. | High, compliance-sensitive | Medium | Recheck `marketing_opt_in` inside the claim query and immediately before Twilio send. When opt-in is revoked, mark open campaign sends as `suppressed_opt_out_at`. Keep sends queue-based so each job checks consent at execution time. Also handle Twilio STOP/21610 even though SMS opt-out is listed out of scope. |

Blocking design changes before implementation: add campaign send status/retry fields, make inbound booking idempotent, avoid ambiguous numeric replies across multiple campaigns, and move suppression/eligibility checks into transactional database operations where possible.
