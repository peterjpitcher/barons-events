**Repo Reality Map**

Ground truth: auth is now a dual-layer system. Supabase JWT is the first gate, and a custom `app_sessions` record keyed by `app-session-id` is the second gate. Idle/sliding expiry and heartbeat renewal are gone; only 24-hour absolute expiry remains. That fits the repo conventions that make project CLAUDE authoritative, use server actions for mutations, and reserve the service-role client for system operations in [project CLAUDE](/Users/peterpitcher/Cursor/BARONS-BaronsHub/CLAUDE.md#L34), [project CLAUDE](/Users/peterpitcher/Cursor/BARONS-BaronsHub/CLAUDE.md#L44), [workspace CLAUDE](/Users/peterpitcher/Cursor/CLAUDE.md#L36), [workspace CLAUDE](/Users/peterpitcher/Cursor/CLAUDE.md#L143), [workspace CLAUDE](/Users/peterpitcher/Cursor/CLAUDE.md#L159).

**Lifecycle**

1. Creation: `signInAction()` validates credentials, checks Turnstile, enforces email+IP lockout, signs into Supabase, clears lockout on success, then creates the custom app session and sets the `app-session-id` cookie. If custom session creation fails, it immediately signs Supabase back out to avoid a JWT-without-app-session state. See [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts#L66), [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts#L145), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L36), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L22).

2. Session record semantics: `createSession()` uses the service-role client, generates a UUID, sets `created_at`, `last_activity_at`, and `expires_at = now + 24h`, stores `user_agent`/`ip_address`, and evicts oldest sessions when a user already has 5 active sessions. Cookie settings are `httpOnly`, `sameSite: "strict"`, `secure` in production, `path: "/"`, `maxAge: 24h`. See [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L36), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L22), [admin.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/admin.ts#L5).

3. Validation on requests: middleware runs on all non-public, non-`/api/*` routes, validates the Supabase JWT with `supabase.auth.getUser()`, requires the `app-session-id` cookie, calls `validateSession()`, and then verifies `session.userId === user.id`. Missing, invalid, expired, or mismatched app sessions redirect to `/login`; invalid/mismatched ones also clear the custom cookie. See [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L13), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L172), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L214), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L244), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts#L281), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L79).

4. Absolute expiry behavior: `validateSession()` only checks `expires_at`. If expired, it asynchronously deletes the row and logs `auth.session.expired.absolute`, then returns `null`. There is no idle check and no renewal path. `lastActivityAt` is still returned, but not enforced. See [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L92), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L107), [audit-log.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/audit-log.ts#L53).

5. User resolution in server components: after middleware succeeds, `getCurrentUser()` prefers the verified `x-user-id` header and loads the app profile from `users`; `layout.tsx` now renders `AppShell` directly, with no idle-timeout provider in the tree. See [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts#L48), [layout.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/layout.tsx#L157).

6. Destruction: `signOutAction()` gets the current Supabase user, destroys the custom session row if the cookie exists, clears the cookie client-side, logs `auth.logout`, signs out of Supabase, and redirects to `/login`. Role changes also destroy all sessions, and password reset destroys all sessions before issuing a replacement app session. See [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts#L177), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts#L58), [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts#L311).

7. Cleanup: the cron route is bearer-protected by `CRON_SECRET`, logs invocation/completion, and calls `cleanupExpiredSessions()`, which bulk-deletes `app_sessions` where `expires_at < now` and prunes `login_attempts` older than the lockout duration. See [cleanup-auth route](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/api/cron/cleanup-auth/route.ts#L12), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L144), [session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts#L157).
