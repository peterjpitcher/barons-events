Two high findings remain exploitable.

1. `SEC-001` High: `cancelBookingAction()` still authorizes against a caller-supplied `eventId`, not the booking’s real `event_id`. Exploit: a `venue_manager` who knows any victim `bookingId` can invoke the action with `cancelBookingAction(victimBookingId, attackerOwnedEventId)`. The server only checks `venue_id` on the supplied event, then `cancelBooking()` updates `event_bookings` by `bookingId` alone through the service-role client. That allows cross-venue booking cancellation. The same spoofed `eventId` is then written into the audit log and cache revalidation path, which misattributes the action and helps hide the real target. References: [src/actions/bookings.ts#L167](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts#L167), [src/lib/bookings.ts#L83](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/bookings.ts#L83).

2. `SEC-002` High: Turnstile on public bookings is still bypassable by omission. Exploit: a bot can submit the booking action without any Turnstile token. The client sends `undefined` when the hidden field never appears, the schema explicitly makes `turnstileToken` optional, and `verifyTurnstile()` returns `true` when `token` is missing. Result: anonymous spam bookings still succeed behind only the IP limiter, which rotating proxies can evade. References: [src/app/l/[slug]/BookingForm.tsx#L49](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/l/[slug]/BookingForm.tsx#L49), [src/actions/bookings.ts#L19](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts#L19), [src/lib/turnstile.ts#L6](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/turnstile.ts#L6).
