# Assumption Breaker Report v4

## Summary

v4 is **not ready to implement**.

It resolves several v3 blockers, especially the SOP partial-index collision, rejected proposal CHECK, attachment INSERT RLS, and the basic RPC transaction model. But the new concrete SQL exposes fresh blockers: the multi-venue RPCs under-authorise users, the draft RPC inserts an invalid `planning_items.status`, the proposal RPC is ordered before the schema can accept proposal rows, and the audit CHECK still misses a live trigger action.

Verdict: **NEEDS V5**.

## PART B — Concrete SQL Issues

**AB-V4-001 — BLOCKER: multi-venue RPC authorisation is weaker than the role model.**  
The behaviour text says to verify `canManageEvents` for every venue: [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:560). The SQL only rejects venue-scoped office workers targeting a different venue: [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:622), [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:764). It does not reject `executive` or no-venue `office_worker`, both non-writers under the repo role model: [roles](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/roles.ts:20). Because the RPCs are service-role-only, RLS will not save this: [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:693).

**AB-V4-002 — BLOCKER: draft RPC inserts invalid `planning_items.status = 'open'`.**

Existing event planning creation uses `planned`: [events](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/events.ts:560).

`audit_users_sensitive_columns()` writes `user.sensitive_column_changed`: [migration](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260414160001_users_sensitive_column_audit.sql:17), [migration](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260414160001_users_sensitive_column_audit.sql:21). ## What Appears Newly Sound in v4

- Per-venue children avoid `idx_planning_tasks_sop_idempotent`: child `sop_template_task_id` is `NULL`, and the existing index only includes non-null rows.
- Per-venue masters correctly set `cascade_sop_template_id = v_tmpl_id`: [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1157).
- SOP RPC bypass scope is sound: it uses transaction-local `set_config(..., true)`: [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1130), and the guard checks `cascade_internal_bypass()`: [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1382).
- SOP idempotency correctly skips a per-venue rerun because at least the master has `sop_template_task_id IS NOT NULL`: [spec](/Users/peterpitcher/Cursor/BARONS-BaronsHub/docs/superpowers/specs/2026-04-17-client-enhancement-batch-design.md:1112).
- The idempotency claim pattern for multi-venue RPCs is transactionally implementable. `INSERT ... ON CONFLICT DO NOTHING RETURNING id` will serialise on the unique key, and failures roll back the claim.
- Attachment INSERT RLS now matches the no-venue office-worker role model.

## Recommendation

**NEEDS V5.**

Minimum v5 fixes: add `user.sensitive_column_changed` to Wave 0, fix multi-venue RPC role checks, change draft planning item status to `planned`, move event status/null relaxation before the proposal RPC, define approval-time planning item creation, and make the Wave 4 caller migration concrete.
