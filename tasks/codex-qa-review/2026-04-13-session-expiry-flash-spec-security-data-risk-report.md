**Findings**
- `Medium`: `redirectedFrom` sanitization is not sufficient. [src/app/login/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx:21) and [src/actions/auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:67) reject absolute URLs and `//...`, but still accept backslash forms like `/\evil.example`, which URL normalization can treat as an external redirect. Exploit scenario: an authenticated user is sent to `/login?redirectedFrom=/%5Cevil.example`; the login page or sign-in action redirects them off-site.
- `Medium`: The dual-layer session model is sound only when the request goes through middleware. [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:196) does the right thing: server-validated Supabase user, then `app-session-id`, then user/session binding. But [src/lib/auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:48) can bypass that model by trusting `x-user-id` or falling back to Supabase-only `getUser()`, and [src/app/login/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx:41) already uses it on a public route. Exploit scenario: `app-session-id` is expired but the Supabase JWT is still valid, so `/login` can misclassify the user as authenticated and bounce them away from re-auth; any future middleware-excluded route using `getCurrentUser()` would inherit the same weakness.
- `Medium`: The flash can expose real sensitive data, not just shell chrome, if the stale tab was already on a sensitive page. Examples: [src/app/customers/CustomersView.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/customers/CustomersView.tsx:66) shows names, mobile numbers, emails, and marketing flags; [src/app/customers/[id]/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/customers/[id]/page.tsx:59) shows full profile and booking history; [src/app/events/[eventId]/bookings/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/events/[eventId]/bookings/page.tsx:119) shows booking names, mobiles, and emails. Exploit scenario: someone with access to an unlocked machine or shared screen sees stale PII before redirect.
- `Low`: A client-side monitor built on `supabase.auth.getUser()` alone is not an auth control. [src/lib/supabase/client.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/client.ts:12) keeps Supabase auth refreshed, while [src/lib/auth/session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:7) enforces a separate 24h app-session. Exploit scenario: the client sees “valid Supabase session” and leaves stale UI in place, but the next protected request is still rejected by [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:228). That is mainly a UX/confidentiality-window issue, not a server-side auth bypass.
- `Low`: Auth failure handling is inconsistent, but I do not see a direct auth bypass from it. [src/actions/sop.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/sop.ts:22) throws, [src/actions/bookings.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/bookings.ts:171) returns errors, and [src/app/error.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/error.tsx:17) renders a generic boundary. The security impact is mostly minor disclosure/behavior inconsistency: `sop.ts` can reflect caught `error.message`, while the global error page does not expose raw messages.

**Assessment**
- Current server-side session security is good on protected page requests. The middleware path is fail-closed, enforces the 24h absolute timeout, clears invalid cookies, and the user/session binding check at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:244) is the right session-fixation defense.
- Cookie settings in [src/lib/auth/session.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/session.ts:22) are strong: `HttpOnly`, `SameSite=Strict`, host-only, `Secure` in production, 24h `maxAge`. I did not find a classic session fixation issue in the current flow.
- Middleware redirect construction at [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:202) is safe against open redirect: it always redirects to same-origin `/login` and only carries internal path/query state forward.

**Fix Direction Guardrails**
- Treat any client-side session check as UX-only. `supabase.auth.getUser()` over HTTPS is not meaningfully interceptable by a passive attacker, but it can be spoofed on a compromised client (XSS, extension, monkey-patched fetch) and it does not know whether `app-session-id` is still valid.
- Do not try to read `app-session-id` from JS. It is `HttpOnly` by design. A JS-readable mirror cookie would be tamperable and should not be used for auth decisions.
- If you add a client-side monitor, have it call an app-owned, same-origin endpoint that exercises the same server truth as middleware, or perform a lightweight same-origin probe to a protected route. That is the only way to account for RC4 correctly.
- Using `window.location.href` for the redirect is safe if the target is constructed locally from a fixed same-origin `/login` plus `location.pathname + location.search`. It is not safe if it ever accepts an arbitrary redirect target. Harden `sanitizeRedirect` to reject backslashes as well as `//`.

**Bottom line**
The current fix boundary is server-side middleware, and that part is solid. The main security risk in the flash bug is stale-data exposure on already-open sensitive pages, plus the risk of implementing the client monitor against the wrong session signal.
