**Findings**
1. High: the `-365` lookback widens the planning backlog, not just calendar history. `lowerBound` is shared by both the `planning_items` and `events` queries, so this change expands `planningItems` as well as events ([index.ts:475](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:475), [index.ts:520](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:520), [index.ts:552](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:552)). Those widened `planningItems` drive alert counts and the todos view ([index.ts:572](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:572), [index.ts:578](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:578), [planning-board.tsx:497](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:497), [planning-todos-by-person-view.tsx:75](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-todos-by-person-view.tsx:75)). If there is open backlog older than 30 days, this will start showing it as overdue everywhere, which is broader than “show past items on the calendar.”

2. High: the default `/planning` experience becomes archive-heavy before the user ever opens calendar. The page fetches `listPlanningBoardData` once ([page.tsx:25](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/planning/page.tsx:25)), and `PlanningBoard` defaults to `board` mode with events visible ([planning-board.tsx:102](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:102), [planning-board.tsx:108](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:108)). Board and list both consume the same combined entry set ([planning-board.tsx:249](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:249), [planning-board.tsx:428](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:428), [planning-board.tsx:487](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:487)). The past bucket renders whenever any past row exists and sorts oldest-first ([planning-board.tsx:241](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:241), [planning-board.tsx:226](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:226)); the list is also oldest-first ([planning-list-view.tsx:41](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-list-view.tsx:41), [planning-list-view.tsx:61](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-list-view.tsx:61)). So year-old completed events can sit ahead of genuinely overdue work in the operational views.

3. Medium: `365` is arbitrary and still does not mean “all.” Calendar navigation is open-ended month-to-month ([planning-calendar-view.tsx:71](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-calendar-view.tsx:71), [planning-calendar-view.tsx:101](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-calendar-view.tsx:101)), but the server silently cuts history off at one year ([index.ts:475](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:475)). That creates a hidden cliff: some history is available, older history just disappears, and the UI gives no indication why. If the requirement is truly “show all items,” this does not satisfy it; if the requirement is “show recent history,” one year on every view is probably too much.

4. Medium: indexes help the query, but not the payload and client work this adds. The DB has indexes on `planning_items.target_date` and `events.start_at` ([20260223120000_add_planning_workspace.sql:53](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20260223120000_add_planning_workspace.sql:53), [20250218000000_initial_mvp.sql:69](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250218000000_initial_mvp.sql:69)), but `listPlanningBoardData` still loads nested task relations for every matching planning item ([index.ts:480](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:480), [index.ts:500](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:500)) and hands the full result to a client component ([planning-board.tsx:1](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:1), [planning-board.tsx:30](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:30)). The client then filters/groups/sorts the arrays multiple times ([planning-board.tsx:112](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:112), [planning-board.tsx:152](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:152), [planning-board.tsx:249](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-board.tsx:249)). Since calendar initially opens on the current month, most of that extra history is paid for before it is visible.

**Open Questions**
- The stated goal says “all items,” but planning items still exclude `done` and `cancelled` by default ([index.ts:531](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:531)). That makes the new history asymmetric: completed events appear, completed planning items do not.
- I would not solve that by widening this shared loader again. Server task alerts ignore parent item status ([index.ts:577](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/planning/index.ts:577)), and the default todos view only filters on task status/date ([planning-todos-by-person-view.tsx:77](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-todos-by-person-view.tsx:77), [planning-todos-by-person-view.tsx:95](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/planning/planning-todos-by-person-view.tsx:95)). Closed planning items with leftover open tasks would leak into alerts and todos.
- I didn’t find targeted tests covering `listPlanningBoardData` or these planning view fan-outs, so this change currently relies on manual behavior checks.
