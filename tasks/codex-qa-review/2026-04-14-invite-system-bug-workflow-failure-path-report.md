**Findings**
- `Critical` The resend control is an actual nested `<form>` inside each row `<form>` on `/users`, in both mobile and desktop rows: [users-manager.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/users/users-manager.tsx:178), [users-manager.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/users/users-manager.tsx:280), [resend-invite-button.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/users/resend-invite-button.tsx:34). I verified the SSR HTML parse: on a hard-loaded page the inner form collapses into the outer form, so clicking `Resend invite` submits `updateUserAction`, and the row’s `Save changes` button can end up outside any form.
- `Critical` The email/confirm flow mixes two different Supabase patterns. The app emails raw Supabase `action_link`s from `generateLink()` ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:121), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:238), [notifications.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:379)), but `/auth/confirm` is written for the SSR `token_hash` flow ([route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:14)). Supabase’s SSR guide says server routes should receive `.../auth/confirm?token_hash=...`; `ConfirmationURL`/`action_link` is the `/auth/v1/verify?token=...&redirect_to=...` pattern. This repo never converts `hashed_token` into the SSR link, and there is no integration test for the second hop.
- `High` `inviteUserAction` creates the auth user and upserts `public.users` before attempting email delivery ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:147), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:160)). If Resend is missing/misconfigured, the admin gets an error but the pending user already exists.
- `High` If `generateLink()` returns a `userId` but no `action_link`, invite flow silently skips email and still returns `Invite sent.` ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:138), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:160), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:177)).
- `High` Any failure after email send but inside the post-link `try` block can trigger rollback delete of the auth user ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:168), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:181)). That means an invite email can be sent and then invalidated.
- `Medium` Invite form validation is partially invisible. The form only renders field errors for `email` and `role` ([users-manager.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/users/users-manager.tsx:60), [users-manager.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/users/users-manager.tsx:117)), but server validation can also fail on `fullName` and `venueId` ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:84)). Because the effect suppresses toasts when `fieldErrors` exist ([users-manager.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/users/users-manager.tsx:63)), those failures look like a no-op.
- `Medium` `/auth/confirm` failure reasons are mostly invisible. It redirects with `?error=missing_token|invalid_token|server_error` ([route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:31), [route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:57), [route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:67)), but `/login` only renders UI for `reason=session_*`, not `error=*` ([login/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx:32), [login/page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/login/page.tsx:62)).

**Workflow 1**
1. `/users` load can fail before any invite: missing auth redirects to `/login`, non-planner redirects to `/unauthorized`, missing/invalid env can throw, and `listUsersWithAuthData()` can fail page render ([page.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/users/page.tsx:14), [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:48), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/users.ts:29), [env.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/env.ts:3)).
2. Filling the invite form has no client validation; everything is server-side because the form uses `noValidate` ([users-manager.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/components/users/users-manager.tsx:81)).
3. Clicking `Send invite` can be intercepted by middleware before the action if the admin session expired; on `/users` that means redirect to `/login`, not an `ActionResult` toast ([middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:201), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:216)).
4. `inviteUserAction` can fail early on auth/profile/env: `getCurrentUser()` returning `null` redirects to `/login`; bad role returns a friendly failure; missing env can throw uncaught ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:95), [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth.ts:82), [admin.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/supabase/admin.ts:14)).
5. `generateLink()` failures are only partly user-friendly: `429` gets a specific message, any other returned error becomes generic, and a thrown rejection is uncaught ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:130)).
6. If Supabase returns `userId` but not `action_link`, the admin sees success and the DB row is written, but no email is sent. Silent failure.
7. `public.users.upsert(...)` can fail on unique email, invalid venue FK, or service-role issues; the catch tries `deleteUser(userId)`, but ignores returned delete errors and always returns the same generic message ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:148), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:181), [initial_mvp.sql](/Users/peterpitcher/Cursor/BARONS-BaronsHub/supabase/migrations/20250218000000_initial_mvp.sql:34)).
8. Email send is a hard failure from the admin’s perspective, but not atomic. Missing `RESEND_API_KEY` or a Resend send error makes `sendInviteEmail()` return `false` ([notifications.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:380)), and the action returns an error after the auth user and `public.users` row already exist ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:162)).
9. The emailed link is the raw Supabase verify URL. That makes invite acceptance dependent on the Supabase `/verify` hop, link scanners, and redirect config. The repo does not test that path end-to-end.
10. After the email click, `/auth/confirm` only works if it gets `token_hash` or `code`. Missing token becomes `/login?error=missing_token`; verify failure becomes `/login?error=invalid_token`; exception becomes `/login?error=server_error` ([route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:28)).
11. Successful invite/recovery verification redirects to `/reset-password`, which is public and not middleware-protected ([route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:62), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:13)). Password reset success does not auto-login; it shows a success card with a manual login link ([reset-password-card.tsx](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/reset-password/reset-password-card.tsx:46)).

**Workflow 2**
1. The pending-user UI itself can be wrong: `listUsersWithAuthData()` treats a `public.users` row with no auth match as `Pending`, so the resend button can appear for a user with no auth invite to resend ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/users.ts:69)).
2. On a hard-loaded `/users` page, clicking `Resend invite` submits the outer update form, not `resendInviteAction`. Likely visible result: no resend loading state, no resend toast, and possibly a `User updated.` toast from the row action.
3. If `resendInviteAction` does run, it can fail at `getUserById`, already-confirmed guard, `generateLink`, missing `action_link`, or Resend send. All return friendly failures except thrown rejections and the post-send `hashEmailForAudit()` call, which are uncaught ([users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:227), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:238), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:255), [users.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/users.ts:270)).
4. Resend has a TOCTOU gap: it checks `email_confirmed_at` by `userId`, then generates/sends using the client-supplied `email` with no cross-check that the auth user still matches that email. There is no idempotency or lock, so double-clicks/concurrent resends can send multiple emails.

**Workflow 3**
1. The emailed URL is a Supabase verify URL, not a direct app URL. The repo’s own tests model it as `.../auth/v1/verify?token=...&type=invite&redirect_to=.../auth/confirm` ([invite.test.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/auth/__tests__/invite.test.ts:20)).
2. `/auth/confirm` is public, so middleware does not block it or `/reset-password` ([middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:13), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:158)).
3. The confirm route assumes the browser arrives with `token_hash` or `code`. If Supabase lands there with only fragment tokens, the server route cannot read them. If it lands with `code` but no `type`, this route exchanges the code and then falls through to `/`, not `/reset-password` ([route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:15), [route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:48), [route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:66)).
4. That `/` fallback is bad for invite acceptance because `/auth/confirm` never creates an `app-session-id`; if the next hop is protected, middleware sees a valid Supabase user but no app session and redirects to `/login?reason=session_missing` (or `session_expired` / `session_mismatch` if a stale cookie exists) ([route.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/app/auth/confirm/route.ts:26), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:216), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:235), [middleware.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/middleware.ts:261)).
5. If the user reaches `/reset-password` without a valid Supabase session, submitting the form just yields the generic “Request a new reset link” style error; the declared `missing-token` / `expired` states are never actually returned ([auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:33), [auth.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/actions/auth.ts:299)).

**Other requested checks**
- Existing auth account: there is no precheck in `inviteUserAction`. The outcome is delegated to Supabase `generateLink()` and untested here. If Supabase errors, the admin gets generic failure. If Supabase returns an existing `userId`, this code will upsert and potentially overwrite that user’s role/venue/full name. That part is inference from the code path, not something the repo tests.
- Token expiry: the app has no dedicated expired-invite UX. It collapses token failures into `/login?error=invalid_token`, which the login page does not display. Also, the invite email says “valid for 7 days” ([notifications.ts](/Users/peterpitcher/Cursor/BARONS-BaronsHub/src/lib/notifications.ts:392)), but I see no repo config enforcing that.
- `RESEND_API_KEY` missing: hard failure, not soft. The admin sees an error, but the new invite path has already created the user records.
- `useActionState` / toast edge cases: repeated success/error should still toast. The real bug is structural: the resend component may never receive state at all because its form does not own the submit in the SSR DOM.
- I ran `npx vitest run src/lib/auth/__tests__/invite.test.ts`; it passes, but it only covers mocked action paths, not `/auth/confirm`, middleware redirects, or real browser form parsing.
